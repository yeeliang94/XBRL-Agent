Product Overview

XBRL Agent

An AI assistant that reads a company's audited financial statement and fills in the official Malaysian regulatory filing template — automatically. Written for two readers at once: the product/business reader and the engineer. Use the switch to hide the technical detail if you don't need it.

Show technical details

In a nutshell

Malaysian companies must file their results with the regulator (SSM) in a rigid, machine-readable format called XBRL. Doing it by hand means copying hundreds of numbers from a PDF into a strict Excel template — slow, and easy to get wrong. This tool does that copying for you and checks its own work.

In plain terms

You upload the financial statement (a PDF or Word document). The tool figures out where each number lives, drops it into the right cell of the official template, cross-checks that the numbers add up, and hands you back a finished, downloadable Excel file — with a full audit trail of what it did and why.

Technically

A standalone extraction agent for Malaysian financial-statement PDFs. Multiple AI agents run concurrently under a coordinator, each responsible for one statement; their results merge into one workbook and a suite of deterministic cross-checks validates consistency. Every figure is stored as a structured "fact" in a database (keyed to its XBRL concept), so verification, review, scoring, and export all read from one canonical source rather than re-parsing spreadsheets.

What it produces

The five face statements

Balance sheet (SOFP), profit & loss (SOPL), comprehensive income (SOCI), cash flows (SOCF), and changes in equity (SOCIE).

The supplementary notes

Five notes templates — corporate info, the list of notes, and detailed disclosure tables that sit behind the headline numbers.

Two standards, two levels

Works for both MFRS and MPERS accounting standards, and for Company-only or Group (consolidated) filings.

A checked, downloadable workbook

A filled Excel template where the totals still recalculate live, plus a report of every consistency check and anything flagged for a human.

The big picture

One document goes in; a filled, validated template comes out. In between, the work fans out to several AI agents working in parallel, then converges again for merging and checking.

The agents in the middle run in parallel — one per statement, plus the notes. Everything converges on the facts database, which every later step reads from. Runs on your machine (or a private server); only the document text is sent to the chosen AI model.

What happens at each stage

Follow one document from upload to download. Optional stages are marked; two stages run many agents at once.

1

Upload & configure

In plain terms

You drop in the financial statement and answer a few setup questions: which accounting standard (MFRS or MPERS), whether it's a Company or Group filing, which statements to extract, and whether to include the notes. Word documents are accepted too — they're converted to a PDF behind the scenes so everything downstream is identical.

Technically

POST /api/upload stores the document as uploaded.pdf in a per-run session directory and inserts a draft runs row (so even abandoned uploads appear in History and the upload is instantly shareable as /run/{id}). A .docx is converted at the door via LibreOffice/Word COM (ingest/word_convert.py); a failed conversion is a clean 422, never a crash. Config (filing_standard, filing_level, statement variants, notes selection) threads end-to-end from RunConfig onward and picks the correct template under XBRL-template-MFRS/ or -MPERS/, Company/ or Group/.

2

Scout the document Optional

In plain terms

Before the heavy lifting, a lightweight "scout" pass skims the whole document to build a map: which page each statement starts on, where the notes are, what currency and units the figures use (thousands? millions?), and the company name and reporting periods. This map is advice, not law — it just gives the extraction agents a running start so they don't have to hunt.

Technically

The scout produces an Infopack: page hints (face_page, note_pages), a notes_inventory with sub-note hierarchy, per-face-line references, and top-level context (entity_name, periods, currency, scale_unit, consolidation_level). Crucially these are soft — agents may read any page; there is no page-restriction enforcement. Every scout-observed field is rendered into prompts with explicit "VERIFY before using" framing (a wrong scale_unit would silently 1000×-inflate every value). Triggered via the Scout button (POST /api/scout), recorded once per run.

3

The coordinator divides the work

In plain terms

A manager component splits the job into independent pieces — one per statement, plus the notes — and launches them all at the same time rather than one after another. It also opens an audit record so that whatever happens next (success, failure, or a user hitting "Stop"), the run is always accounted for.

Technically

run_multi_agent_stream creates the audit row before validation, then wraps the whole body in try/except/finally so every exit path leaves a terminal status (completed, completed_with_errors, failed, aborted) and never a stuck running. It fans N extraction agents and M notes agents onto the event loop, threading run_id + db_path into the tools so writes project live into run_concept_facts.

4

Extraction agents read the statements Runs in parallel

In plain terms

This is the core work. Each agent is an AI given one statement to fill. It reads the relevant PDF pages, matches each line it sees ("Total assets", "Revenue", "Cash and cash equivalents") to the correct row in the template, and writes the number in. It can double-check its own arithmetic with a calculator, look up what a template concept means, and it's told never to fudge a number into a catch-all row just to make a total balance — if something doesn't reconcile, it leaves it and flags it.

Technically

Each agent uses view_pdf_pages, a batched calculator, a concept-definition lookup, and workbook-writing tools. Two hard guardrails: (1) rows whose column-A cell carries the section-header fill are XBRL-abstract and refuse writes; (2) prompts forbid plugging residuals into "Other…"/catch-all rows to satisfy a total. Values land as facts in run_concept_facts, keyed by concept_uuid + period + entity scope. Agents are capped at 40 iterations (below PydanticAI's silent 50-request limit); every agent — including failures — writes a full request/response trace to disk. On Group filings the agent fills both consolidated and company columns.

5

Notes agents fill the disclosure tables Runs in parallel

In plain terms

Alongside the headline statements, separate agents handle the supplementary notes — the detailed breakdowns (property values, borrowings, related-party dealings, accounting policies). These can be rich, formatted tables, so the notes are written as lightly-styled content that preserves the structure of the original, and a coverage checklist tracks that every note in the document actually landed somewhere.

Technically

Notes fill MBRS templates 10–14 (MFRS) / 11–15 (MPERS). Discovery is PDF-first (scout inventory), then pure LLM judgement — no deterministic matching or synonym dictionary. Sheet 12 (the list of notes) fans out into model-aware sub-agents. Cells are sanitized HTML stored in notes_cells (the DB is source of truth; the xlsx download re-overlays from it at stream time). Styling travels through a constrained format_ops sidecar so the content channel stays style-free. A post-run coverage checklist reconciles every top-level note against where its content landed and can tip the run to completed_with_errors if a note is missing.

6

Merge into one workbook

In plain terms

All the separate pieces are stitched back together into a single Excel file that matches the official template exactly — with the totals left as live formulas so they still recalculate when someone opens or edits the file.

Technically

_export_canonical_workbooks re-renders each succeeded statement from run_concept_facts (via the exporter) and workbook_merger.py combines them. mark_run_merged writes the download pointer immediately after a successful merge so the file survives even if later steps crash. All workbook saves route through one atomic tmp-file+os.replace helper to survive concurrent readers. Cross-sheet total formulas stay live; the exporter never overwrites the formula cells.

7

Cross-checks validate the numbers

In plain terms

The tool now audits its own output with a battery of accounting sanity checks — does the balance sheet actually balance? Does profit flow correctly into equity? Does the cash-flow statement tie back to the cash on the balance sheet? Each check passes, fails, or is flagged, and you see the results fill in live as they run.

Technically

A cross-check framework reads facts (default XBRL_FACT_BASED_CHECKS=1) rather than re-opening workbooks, scoped variant-precisely by template ID. Checks declare which standard/level they apply to (e.g. sore_to_sofp_retained_earnings is MPERS-only). On Group filings they run twice — Group columns then Company columns — and report separately. Progress streams as cross_check_start / _result / _complete SSE events so the UI fills rows incrementally.

8

The reviewer pass fixes what it can Automatic

In plain terms

If checks fail, a second, more senior AI "reviewer" investigates the root cause — tracing a wrong total back through the sub-tables to the original PDF — and applies grounded corrections. It can only fix things it can justify from the source; anything it can't resolve it raises as a flag for a human. A one-click "Revert to original" always restores the first extraction. Even on a clean run, a light spot-check gives it a sanity read.

Technically

_run_reviewer_pass investigates failing checks + open conflicts down the face→sub→PDF chain and writes through a guarded apply_fixes tool (the same no-plug/no-abstract guard as extraction). Safety is versioning, not write-gating: snapshot_facts runs once before any write, so revert is one click; the pass then re-exports and re-merges. Scoped to the run's {standard}-{level}- family. Gated by XBRL_AUTO_REVIEW (on) and XBRL_SPOT_CHECK (on) for clean runs. Manual re-review runs async on a dedicated thread, tracked in run_review_tasks.

9

Download & review

In plain terms

You get the finished Excel file, plus a full run page: every statement and its agents, the notes, the cross-check results, a telemetry tab (how long it took, how many tokens), and a review workspace where you can see each figure, its evidence from the PDF, and edit anything by hand. Nothing is a black box — every number traces back to a page.

Technically

The run-detail surface is one tabbed view (RunDetailView.tsx): Overview · Agents · Notes · Cross-checks · Telemetry · Review · Values. The Review/Values tabs read the canonical fact model; evidence citations link to the PDF viewer. Downloads recompute in Excel (live formulas). Verbatim transcripts live on disk and are served on demand, not stored in SQLite.

Beyond a single extraction

The same fact model powers several surrounding capabilities.

mTool fill

Plain: pushes the extracted figures straight into SSM's official "mTool" filing template so an operator can validate and generate the XBRL without hand-copying.

Technically

Offline zip/XML surgery (mtool/offline_fill.py, stdlib-only, no Excel), so it runs server-side and in the cloud. Values emitted verbatim; a button + modal, not a tab.

Evals & benchmarks

Plain: measures how accurate the tool is by scoring a run against a human-verified "gold" answer, and tracks that accuracy over time and across many documents.

Technically

Gold stored as facts (same shape as run facts); grading is a set-join on concept_uuid — exact, not a brittle cell-diff. Repeats/consistency, suites, and trend charts sit on top (eval/).

Telemetry & audit trail

Plain: every run records timing, token usage, tool activity, and a full transcript of what each agent did — so a failure is debuggable and a success is defensible.

Technically

Per-turn figures derived from cumulative usage deltas; exact timing + tool calls persisted to run_agent_turns. Traces (incl. failures) written to disk per statement.

Accounts & access

Plain: the web app sits behind email + password login, with an admin who manages who's allowed in. No self-signup.

Technically

Server-side revocable sessions, 15-min sliding idle timeout, per-(email,IP) lockout. Every /api/* route is gated (auth/ package). Admin role is the privilege boundary, enforced server-side.

!
A deliberate design choice worth knowing. The notes pipeline uses pure AI judgement — no hard-coded dictionary of "this label means that row." That keeps it flexible across wildly different report layouts, but it's also why the tool leans so heavily on cross-checks, the reviewer pass, and the human review workspace to catch mistakes rather than assuming the first pass is always right.

Glossary

The acronyms, once, in plain language.

XBRL
eXtensible Business Reporting Language — a standard, machine-readable format for financial data. The end product the regulator wants.
SSM
Suruhanjaya Syarikat Malaysia — the Companies Commission of Malaysia, the regulator you file with.
MBRS
Malaysian Business Reporting System — SSM's platform and template set for XBRL filings.
MFRS / MPERS
The two Malaysian accounting standards. MFRS is the full standard (larger entities); MPERS is the simpler one for private entities. The tool supports both.
Face statements
The five headline financial statements, as opposed to the detailed notes behind them.
SOFP
Statement of Financial Position — the balance sheet.
SOPL
Statement of Profit or Loss — the income statement.
SOCI
Statement of Comprehensive Income.
SOCF
Statement of Cash Flows.
SOCIE
Statement of Changes in Equity.
Notes
The supplementary disclosures — detailed tables and policies that explain the headline numbers.
Company / Group
Filing level. Company = the single entity; Group = the consolidated parent-plus-subsidiaries view. Group templates carry extra columns.
Agent
One AI worker with a focused job (e.g. "fill the balance sheet"). Many run at once.
Scout
A fast preliminary pass that maps the document so the extraction agents start in the right place.
Cross-check
An automated accounting sanity test (does it balance? does it tie back?).
Reviewer
A senior AI pass that investigates failed checks and applies grounded fixes, or flags them for a human.
Fact
One extracted number stored with its meaning (its XBRL concept), period, and scope — the single source of truth everything else reads from.
mTool
SSM's official desktop template/add-in for producing the final XBRL filing.
