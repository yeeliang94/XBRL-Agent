Executive Summary

Most Important Change

Add a mandatory "evidence ledger before write" discipline to the face-statement prompts. The current prompts say to inspect linked notes, but agents can still rush from face line to workbook write.

Highest Risk Behavior

Catch-all rows remain necessary, but the model needs a stronger precondition checklist before using any "Other" row. This directly targets the shortcut of combining unresolved differences into a single other field.

Best Existing Defense

The repo already has strong guards: no-residual rules, save gates, Sheet-12 coverage receipts, label catalogs, and retry loops. The recommendations should reinforce those instead of replacing them.

Priority Recommendations

P0 Require an Evidence Ledger Before Writing

Current prompts tell agents to read the template, face statement, and linked notes, but there is no explicit "do not write until you have a ledger" rule. This is the most direct fix for agents that jump to a conclusion after seeing the face statement.

  • For each face line with a note reference, the agent should identify the face page, note number, note pages viewed, scale/unit, target template row(s), and whether it is split or rolled up.
  • Only then should it call fill_workbook.
  • Evidence should cite both the face page and note page for note-derived values.

P0 Add Catch-All Row Preconditions

The prompts already say not to plug residuals. The failure pattern you described is more subtle: the model may use "Other" as a convenience bucket before it has exhausted the note and template granularity. Add a checklist that must be satisfied before any catch-all row is used.

  • The PDF itself uses an "Other" or coarse category, or the template has no more specific row.
  • The agent has checked the relevant note continuation pages and sub-schedules.
  • The evidence names the PDF label that justifies the catch-all row.
  • The value is not equal to an unexplained difference between a face total and the filled breakdown.

P1 Add a Retry Playbook to Tool Feedback

The code has retry and save-gate mechanics, but prompts should teach the model how to respond when a tool refuses a save, flags an imbalance, or rejects a label. This should reduce "finish anyway" behavior.

  • For verify_totals failure: re-check scale, sign, current vs non-current section, group vs company column, note continuation, and formula direction before changing labels.
  • For writer warnings or fuzzy matches: use the returned closest labels verbatim, or skip honestly if none fit.
  • For mandatory unfilled rows: search face statement and linked notes first; do not fabricate a zero or "not applicable" unless the PDF explicitly discloses it.

P1 Make Notes Review Depth Observable

The notes prompts are careful about hierarchy and schedules, but the model can still view the first page of a note and write too early. Add a prompt-side requirement to read until the next top-level note heading or end of schedule, then make the evidence reflect the full range.

  • Require source_pages to include every page used for the note, not only the first page.
  • When an inventory range is extended, evidence should say why, such as "continued schedule to Page 34".
  • For Sheet-12 catch-all, require a skip or catch-all reason that names the candidate labels considered.

P1 Expand Malaysia-Specific Prompt Examples

SOFP already has useful Malaysian AFS examples. Similar examples should be added to SOPL, SOCF, SOCIE, and notes. Keep these as examples, not a deterministic dictionary.

  • SOPL: EPF/SOCSO/EIS under employee benefits, directors' fees vs directors' remuneration, unrealised/realised foreign exchange gain or loss, gain or loss on disposal, management fee income or expense.
  • SOCF: deposits pledged to licensed banks, hire purchase and lease principal, interest paid classification, fixed deposits with maturity beyond three months.
  • SOCIE: bonus issue, rights issue, share premium transition, treasury shares, non-distributable reserves, revaluation reserve, foreign currency translation reserve.
  • Notes: distinguish policies from disclosure notes, and distinguish related-party transactions from balances due from or due to related parties on SOFP.

Suggested Wording Blocks

These are draft blocks to insert into existing prompts. They are written to preserve the current architecture and avoid deterministic matching.

1. Add to prompts/_base.md: Evidence Ledger Before Write

=== PRE-WRITE EVIDENCE LEDGER ===

Before your first fill_workbook call, build a brief internal ledger for every
face-statement line you will write:

- face page and face label
- note reference, if any
- note page(s) viewed, including continuation pages
- PDF unit or scale, e.g. RM, RM'000
- target sheet, section, row label, and CY/PY or Group/Company column
- decision: split into component rows, roll up to a coarser template row, or
  write a direct face value because no relevant sub-sheet or analysis row exists

Do not write a note-referenced face line until you have viewed the linked note
or made a documented best-effort search for it. Evidence must cite both the
face page and the note page when a value comes from a note breakdown.

2. Add to prompts/_base.md: Catch-All Preconditions

=== CATCH-ALL ROW PRECONDITIONS ===

Before using any row labelled "Other", "Miscellaneous", "Administrative
expenses", "Other income", "Other expenses", or similar:

1. Confirm the PDF itself discloses the amount as an "other" or coarse category,
   OR confirm the live template has no more specific row for the disclosed item.
2. Confirm you checked the linked note, continuation page, and relevant
   sub-table before deciding no specific row exists.
3. In evidence, name the PDF label that justifies the catch-all row.
4. Never write a value that is merely the difference between a face total and
   the rows you already filled.

If these preconditions are not met, leave the specific rows as found and let
verification flag the gap for human review.

3. Add to prompts/_base.md: Retry Playbook

=== RETRY PLAYBOOK AFTER TOOL FEEDBACK ===

If fill_workbook, verify_totals, or save_result returns errors or warnings,
do not finish immediately. Read the tool feedback and retry only the smallest
necessary correction.

- Label or section error: use read_template and the tool's closest labels,
  then retry with a real label copied from the template.
- Formula or abstract-row refusal: choose a data-entry leaf row below the
  section header; never force a write to the refused row.
- Balance or subtotal mismatch: check scale, sign, CY/PY column, Group/Company
  column, current/non-current section, linked note continuation pages, and
  whether the template formula adds or subtracts the row.
- Mandatory row blank: search the face statement and linked notes before
  entering zero. Use zero only when the PDF explicitly discloses zero.

Do not solve a failed verification by plugging a catch-all row.

4. Add to prompts/_notes_base.md: Complete Note Boundary

=== COMPLETE NOTE BOUNDARY BEFORE WRITE ===

For each note you write, read until you can see the next top-level note heading
or the relevant schedule clearly ends. If the inventory page range ends while a
table, paragraph, or sub-note is still continuing, view the next page or two.

Your source_pages must include every PDF page you relied on, not only the first
page of the note. If you extended beyond the inventory range, mention that in
evidence, e.g. "Pages 31-33, Note 8, schedule continues to Page 33".

5. Add to prompts/notes_listofnotes.md: Catch-All Is Not a Shortcut

=== CATCH-ALL IS NOT A SHORTCUT ===

Use the catch-all row only when the note's topic genuinely has no row in the
seeded label catalog. Do not use the catch-all row to avoid splitting a mixed
note into its proper rows, and do not use it because the first candidate label
was rejected by the writer.

Before routing a note to catch-all:
- check the seeded label catalog for a specific topic row;
- if a writer rejection occurred, retry using one of the closest labels if it
  genuinely fits;
- include a short reason in the coverage receipt naming the candidate topic
  you considered, e.g. "no MPERS row for capital management disclosure".
