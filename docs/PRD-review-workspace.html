Review Workspace — Evidence-Synced Verification

PRD · Draft for review · 2026-05-26 · Owner: William Chen
TL;DR. The post-run review screens report findings but never take you to them. A cross-check says two figures don't reconcile, but you can't click it; a value is shown, but its source PDF page is in a file you'd open by hand; the sheet dropdown makes you go hunting. This PRD reshapes review around one principle — every finding is a link to its evidence — by adding a server-rendered PDF pane, a clickable-finding selection model, and a navigator that replaces the dropdown. We build these as reusable parts and land them on the ConceptsPage (canonical mode) first, then reuse on History detail and the run-complete summary.

1. Problem & background

After a run finishes, a reviewer has to confirm that the extracted figures are correct against the source financial-statement PDF, and resolve anything the cross-checks flagged. Today that work is spread across three disconnected surfaces:

SurfaceFileWhat it does today
Run-complete summaryweb/src/components/ResultsView.tsxSummary metrics, JSON data preview, download buttons.
History run detailweb/src/components/RunDetailView.tsxRun config, agent timelines, notes review, cross-check table.
Canonical modeweb/src/pages/ConceptsPage.tsxConcept tree-grid, template-selector dropdown, ReconciliationQueue side panel.

The five reported pains

  • Reconciliation is a dead end. ValidatorTab.tsx and ReconciliationQueue.tsx list problems but offer no click-through to the cell they're talking about.
  • No source visible. There is no PDF viewer anywhere in review; verifying a value means opening the original PDF by hand and scrolling.
  • Sheet navigation is unfriendly. Switching template sheets goes through a dropdown rather than a fast, always-visible navigator.
  • The side panel feels inert. The ReconciliationQueue surfaces conflicts but its actions don't move you to the issue.
  • Visual clutter. Animations, spacing, and where the content sits could be calmer and more deliberate.
Feasibility: green. The hard parts already exist. The source PDF is retained per run at output/{id}/uploaded.pdf (never cleaned up). tools/pdf_viewer.py::render_pages_to_png_bytes() already renders any page to a PNG, with an LRU page cache in tools/page_cache.py. Every extracted value carries an evidence string like "Page 14, Note 1", and notes cells additionally store a parsed source_pages integer list. The only missing backend piece is an HTTP route that serves a rendered page.

2. Goals & non-goals

Goals

  • Let a reviewer verify any extracted value against its source PDF page without leaving the review screen.
  • Make every flagged finding (cross-check failure, reconciliation conflict) a link that selects the offending value — driving both the data grid and the PDF pane.
  • Replace the sheet/template dropdown with an always-visible navigator.
  • Reduce visual noise — calmer animation, clearer hierarchy, stable content placement.
  • Build the PDF pane and the selection model as reusable parts so the same pattern can later extend to History detail and the run-complete summary.

Non-goals (this PRD)

  • Not building an in-browser Excel preview — we review values via the concept grid + notes editor, and verify against the PDF.
  • Not changing the extraction pipeline, agents, prompts, or the canonical concept model itself.
  • Not unifying all three surfaces into one route in a single release (that is the eventual end-state, reached incrementally).
  • Not adding PDF text search / OCR / highlight-the-exact-number overlays (possible later; see open questions).

3. Users & primary journey

Primary user: the reviewer/preparer checking a filled MBRS workbook before filing. They trust nothing until they've eyeballed it against the audited PDF.

Primary journey (target):

  1. Open a completed run's review workspace.
  2. Glance at the findings strip — "3 cross-checks failed, 1 conflict open."
  3. Click the first failed check → the offending value is selected in the grid and the PDF pane flips to its evidence page.
  4. Eyeball the figure against the page. Either it's right (dismiss the flag) or it's wrong (correct it, or note it).
  5. Use the navigator to step through sheets; the PDF pane follows each selected value.

5. Functional requirements

5.1 PDF page-serving endpoint P0

  • New FastAPI route, e.g. GET /api/runs/{run_id}/pdf/page/{page}.png?dpi=150, returns a rendered PNG of that 1-indexed page.
  • Resolves the PDF from the run's session dir (output/{id}/uploaded.pdf); 404 if absent (e.g. legacy/CLI runs with no stored copy).
  • Reuses render_pages_to_png_bytes() and the existing page_cache; validates 1 ≤ page ≤ N; clamps DPI to a sane range.
  • Served as an image response with cache headers so the browser can cache pages.
Mirror the safety posture of the existing /api/result/{session_id}/{filename} route — the current download whitelist deliberately excludes the PDF, so this is a new, explicit read path scoped to page images, not raw-file access.

5.2 Evidence → page resolution P0

  • A shared parser maps a value's evidence to one or more page numbers. Prefer notes_cells.source_pages (already a clean int list) when present; otherwise regex the evidence string (Page 14, p.42, Pages 19-20, Page 3; Page 4).
  • If multiple pages, the pane shows the first and offers prev/next within the cited set.
  • If a value has no parseable page, the pane shows an empty state with a manual page jumper — it must never break selection.

5.3 PDF pane component P0

  • Reusable React component: given runId + page (and optional cited-page set), renders the page image with prev/next, a page-number jump, and zoom-to-fit / zoom-in.
  • Server-rendered images via <img> — not a browser PDF library (see §6). Inline styles, per gotcha #7.
  • Loading + error states are quiet (no spinner thrash); a failed page render shows a retry, not a crash.

5.4 Selection model — "what's selected" P0

  • A single source of truth for the currently selected value (sheet + row/concept + evidence pages). Setting it drives the grid scroll/highlight and the PDF pane together.
  • Lightweight (component state or a small context/store) — no heavy state-management dependency.

5.5 Clickable findings P0

  • ValidatorTab.tsx cross-check rows become clickable where the check identifies a target concept/row → sets selection.
  • ReconciliationQueue.tsx conflict rows become clickable → sets selection; Resolve/Dismiss stay as-is.
  • Checks that genuinely have no single target (e.g. a whole-statement balance) select the most relevant anchor row and say so.

5.6 Navigator (replace dropdown) P1

  • Always-visible list of sheets/templates (sorted by MBRS slot order, reusing existing sort) replacing the ConceptsPage template-selector dropdown.
  • Per-sheet finding counts (e.g. a red dot with "2") so reviewers triage where to look first.

5.7 Visual cleanup P2

  • Audit existing animations in index.css (pulse-subtle, glyph-pulse, fade-in, slide-down); keep purposeful ones, drop noise in the review context.
  • Stabilize content placement (no layout jumps as panes load); tighten spacing/hierarchy for the three-region layout.

6. Technical design notes

Why server-rendered page images, not a browser PDF library

The obvious alternative is PDF.js / react-pdf in the browser. We reject it for this codebase because: (a) it ships a separate worker file and heavier bundle; (b) it has a history of being finicky on the Windows/enterprise target (same family of trouble as the Tailwind issue in gotcha #7, which is why the frontend uses inline styles); and (c) we already have render_pages_to_png_bytes() proven in the scout, correction, and extraction agents, with caching. Server-rendered <img> is the boring, robust choice.

Reuse, don't rebuild

  • Rendering: tools/pdf_viewer.py::render_pages_to_png_bytes() + tools/page_cache.py.
  • Evidence: notes_cells.source_pages (DB, parsed ints) and the evidence TEXT columns on extracted_fields / run_concept_facts.
  • Sheet ordering: existing sortSheetsBySlot used by the notes review tab.
Page hints are NOT persisted. Scout page_hints (face/note pages) live in-memory during a run and never reach the DB. They are a nice-to-have fallback for values with no evidence page — out of scope for P0. If we want them post-run, that's a one-column migration on runs (schema is version-stepped per gotcha #11), tracked as an open question, not a requirement.

8. Success metrics

  • Primary (qualitative): a reviewer can verify a flagged value against its source page without opening the PDF separately. Validate by watching one real review session per phase.
  • Every failed cross-check / open conflict that has a target concept is clickable and lands on the right row.
  • PDF page loads feel instant on a second view (cache hit).
  • No reliance on the sheet dropdown remains on ConceptsPage after M3.

9. Risks & open questions

#ItemDisposition
1CLI runs don't copy the PDF into the run dir, so no source to serve.Acceptable: feature targets web runs. PDF pane degrades to an empty state for runs without a stored PDF.
2Evidence strings may be missing, vague, or cite the wrong page.Pane never blocks on this — manual page jumper + prev/next always available. Bad evidence becomes visible (a review win), not a crash.
3Some cross-checks have no single target row.Select the most relevant anchor and label it; don't force a false target.
4 openShould we persist scout page_hints for a smarter fallback page?Decide after M2. One-column migration if yes.
5 openDo reviewers want the exact number highlighted on the page, not just the right page?Out of scope for now; revisit if "right page" proves insufficient.
6 openDPI / image-size tradeoff for legibility vs. payload.Start at ~150 DPI; tune against a real statement page.

10. Out of scope

  • In-browser Excel/workbook preview.
  • PDF text layer, search, OCR, or number-level highlighting.
  • Pipeline / agent / prompt / canonical-model changes.
  • A single merged review route (reached incrementally via the "later" phase, not built up front).
