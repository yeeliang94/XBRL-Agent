# SSM Notes Format Investigation (RUN-REVIEW P0-2)

**Status:** Open — awaiting access to a known-good filer-submitted xlsx.
**Owner:** _unassigned_
**Last updated:** 2026-04-26

## Background

The LRI agent's review of the Amway FY2024 MFRS Company run
(`RUN-REVIEW.md` §3.5) claimed that:

> All five notes agents wrote rich HTML directly into column B of the
> notes sheets. The filer's file, however, carries the literal string
> `[Text block added]` in those same cells and stores the real
> narrative in hidden `+FootnoteTextsN` sheets (`+FootnoteTexts0` …
> `+FootnoteTexts4`, `+Elements`, `+Lineitems`).

If true, this is a high-impact ingestion gap — the SSM MBRS validator
would reject our merged xlsx because the visible cells carry HTML
where it expects a placeholder, and there's no hidden footnote sheet
carrying the real narrative.

## Why this is investigation-only

The reviewer's evidence was observational, captured on a Windows
workstation with access to the filer's submitted file. They did NOT
include the SSM validator's actual error output, the SSM template
itself, or the schema spec for the `+FootnoteTextsN` sheet shape.

Searching the codebase confirms there is **zero in-repo evidence** for
this convention:

- `grep -rn "[Text block added]" SSMxT_2022v1.0/` → 0 hits
- `grep -rn "FootnoteText" SSMxT_2022v1.0/` → 0 hits
- `grep -rn "+Lineitems" SSMxT_2022v1.0/` → 0 hits

The SSM template files in `XBRL-template-MFRS/` and
`XBRL-template-MPERS/` ship without hidden sheets and without
placeholder text. So either:

1. The convention is generated by the SSM SUBMISSION TOOL, not the
   templates we ship from, OR
2. The convention is from a different version of the SSM stack, OR
3. The reviewer was looking at a different artefact entirely.

Without ground truth, building a config-flag-gated render mode that
defaults to the unverified shape risks regressing every existing run
in production. **We need a real filer xlsx to introspect first.**

## How to make progress

Run the validator harness against a known-good filer-submitted xlsx:

```bash
python3 scripts/validate_ssm_compatibility.py path/to/filer-submission.xlsx
```

The script reports:
- Whether `[Text block added]` (or similar) appears in any visible cell
- All hidden / very-hidden sheets and their shapes
- A pass/partial/fail verdict on the convention claim

Drop the verdict into a new section below ("Findings"), then decide:

- **Verdict CONFIRMED** → open `PLAN-ssm-notes-output.md`, design the
  render-mode split (config flag, hidden-sheet writer, DB stays
  canonical per gotcha #16). The harness output gives you the exact
  sheet names, header structure, and any other constraints to honour.
- **Verdict PARTIAL or NOT CONFIRMED** → close RUN-REVIEW P0-2 in
  `docs/PLAN-run-review-fixes.md` as "not actionable in this
  checkout — inline-HTML output is appropriate". Update CLAUDE.md
  gotcha #16 if the investigation surfaces any related finding.

## Why we deferred rather than implemented speculatively

Per `docs/PLAN-run-review-fixes.md` Phase 5: we don't ship config
flags whose default depends on an unverified hypothesis. Building
`notes_output_format: "ssm"` and defaulting it on would break
historical runs and give the wrong impression that we know the
convention's shape. The cost of waiting is low (today's inline-HTML
output is "wrong" only against a hypothetical SSM check); the cost
of being wrong is a regression that touches every notes-bearing run.
