XBRL Agent — what happens beneath the surface

A step-by-step walkthrough of how a Malaysian financial-statement PDF turns into a filled SSM MBRS XBRL workbook: every agent, every tool, every automatic guard, every retry budget, every threshold. All facts here are pulled directly from the source under /Users/user/Desktop/xbrl-agent.

Repo: xbrl-agent Branch: main Generated: 2026-05-21 See also: docs/ARCHITECTURE.md
LLM tool exposed to the agent for it to call
LLM call the agent's actual model invocation
auto runs deterministically, no LLM involvement
guard refuses bad writes / catches mistakes
limit retry / iteration / wall-clock cap
async parallel fan-out

§0 The big picture

One coordinator fans out N face-statement agents and M notes-template agents in parallel. Each agent has the same skeleton: read template → view PDF pages → fill cells → verify → save. A merger stitches per-agent workbooks into filled.xlsx. Cross-checks validate consistency across statements. A correction agent then loops to fix failed checks. Everything is streamed live to the UI over SSE.

Pipeline stages emitted as SSE pipeline_stage events

The coordinator pushes one event at every boundary so the UI can label silent dead-zones:

extracting → merging → cross_checking → correcting → re_checking → validating_notes → done

Single source of truth
Every LLM call — scout, face, notes, correction — routes through server._create_proxy_model(). Two modes: proxy (LiteLLM, OpenAI-compatible) or direct (Gemini / OpenAI / Anthropic detected from the model-id prefix). All models run with temperature=1.0 — anything lower causes Gemini 3 through the proxy to fail or loop.

§2 Scout — PDF reconnaissance

The scout is one PydanticAI agent that walks the PDF before any face/notes agent starts. Its job: find the table of contents, classify the filing standard (MFRS vs MPERS), detect each statement's variant (e.g. SOFP CuNonCu vs OrderOfLiquidity), and build a notes_inventory so notes agents know where to look.

Model: SCOUT_MODEL from .env (defaults to a fast Gemini Flash). Per-tool view cap: MAX_VIEW_PAGES = 5.

Tools the scout LLM can call

  • find_toc() → TocCandidate[]LLM tool
    Scans the first _SCAN_WINDOW = 15 pages with regex _TOC_LINE_RE (matches "statement|note|report|director|audit" + dotted leader + page-number), parses TOC entries (English+Malay supported), and side-effect: caches deps.detected_standard via scout.standard_detector.detect_filing_standard.
  • parse_toc_text(text)LLM tool
    For scanned PDFs: the LLM uses vision to read the TOC and passes the text back. Same parser + standard-detection hook as above.
  • view_pages(pages: List[int])LLM tool
    Extracts fitz text + renders PNGs at dpi=200. Hard-capped at 5 pages per call.
  • check_variant_signals(stmt, page_text)LLM tooldeterministic body
    Calls variant_detector.detect_variant_from_signals — a signal-counting scorer. Each registered variant has detection_signals; matches add to score, _NEGATIVE_SIGNALS subtract 2, and an _ABSENCE_BONUS = 3 is given to OrderOfLiquidity when no current/non-current keywords appear. MPERS-aware: filters out non-applicable variants via variants_for_standard.
  • discover_notes(face_text, notes_start_page=None)LLM tool
    Regex _NOTE_REF_RE = r"\bnotes?\s+(\d{1,3})\b" extracts in-text note references, then a heuristic page mapper (_PAGES_PER_NOTE_ESTIMATE = 3, capped at _MAX_NOTE_PAGES = 30) suggests page ranges.
  • discover_notes_inventory(notes_start_page)LLM toolparallel
    Builds the full notes inventory — see vision fallback.
  • save_infopack(infopack_json)LLM toolguard
    Validates against the StatementType registry. Rejects unknown statement keys and NotPrepared variants the LLM hallucinated. Final output of the scout phase.

Automatic helpers (no LLM)

scout/standard_detector.py

detect_filing_standard(text) — lowercase substring scan.

  • _MPERS_KEYWORDS: "mpers", "malaysian private entities reporting standard", "statement of retained earnings"
  • _MFRS_KEYWORDS: "mfrs", "mfrs 101/9/15/16"
  • If both sets hit → returns "unknown".

scout/toc_locator.py

find_toc_candidate_pages() scoring rubric:

  • Literal "table of contents" → score = 10
  • Pattern match (≥ 2 lines) → score = match count
  • Image-only PDF → heuristic fallback pages 2–6

Returns TocCandidate(page, method, score) sorted desc.

scout/notes_discoverer.py

build_notes_inventory(_async) — fast pass:

  • _NUMBERED_HEADER_RE — "4. PROPERTY, PLANT AND EQUIPMENT"
  • _PREFIXED_HEADER_RE — "NOTE 4 - REVENUE"
  • Returns NoteInventoryEntry(note_num, title, page_range, suggested_row_label).
  • If empty AND a vision_model is supplied → falls through to vision pass.

Vision fallback for scanned PDFs

scout/notes_discoverer_vision.py · _vision_inventory()

  • parallel Pages chunked via _chunk with BATCH_SIZE = 8, OVERLAP = 1.
  • Rendered at _DPI = 150 (lower than face agents' 200, since this is a discovery pass).
  • Batches run under asyncio.Semaphore(MAX_PARALLEL = 5).
  • limit Per-batch one retry on any exception with 0.5 s backoff.
  • LLM returns _VisionBatch (Pydantic structured output).
  • _merge_and_stitch() derives each non-terminal note's last_page from the next note's first_page - 1; trailing note's last_page is the LLM value, clamped to notes_end.
Gotcha #13 — Page hints are soft
The scout produces page_hints = {face_page, note_pages} which are passed to face/notes agents as suggestions. There is no allowed_pages enforcement. view_pdf_pages only validates 1 ≤ page ≤ N. Don't re-introduce page-restriction logic — tests/test_page_hints.py pins this with negative assertions.

§3 Face extraction agents — one per statement

For each statement (SOFP, SOPL, SOCI, SOCF, SOCIE), the coordinator spawns one PydanticAI agent built by extraction.agent.create_extraction_agent(statement_type, variant, …). System prompt = prompts/_base.md + prompts/{stmt}.md (+ _group_overlay.md on Group filings).

Per-agent dependencies (ExtractionDeps)

pdf_path / template_path / filled_pathStatic for the run.
filing_level, filing_standardThreads MFRS/MPERS and Company/Group everywhere downstream.
turn_counter, save_attemptsMutable across the agent loop; used by the save-gate.
last_verify_resultSave-gate state. Cleared by fill_workbook.
template_fieldsCached read_template output so repeated calls don't re-open the xlsx.

Tools the face-agent LLM can call

  • read_template()LLM tool
    Reads template via tools.template_reader.read_template. Output via _summarize_template: each cell is printed as coord (row N): label [ABSTRACT (section header — do not write)] / [DATA_ENTRY] / [FORMULA: =…]. The [ABSTRACT] tag comes from the template's TemplateField.is_abstract flag — the agent learns at first glance which rows are headers it must not write to.
  • view_pdf_pages(pages: List[int])LLM tool
    Renders to PNG bytes at dpi=200 via render_pages_to_png_bytes, parallelised with ThreadPoolExecutor(max_workers=min(len(pages), 8)). Validates 1 ≤ page ≤ total_pages. No allowed_pages filter.
  • fill_workbook(fields_json)LLM toolguards
    Two modes:
    1. Label-match mode: field_label + section + col + value + evidence. Internal fuzzy match resolves the row.
    2. Row-coordinate mode: row + col for cases where the agent already knows the row number.
    After every successful call it clears deps.last_verify_result so the agent must re-verify before saving. Returns warnings verbatim (e.g. double-booking detection).
  • verify_totals()LLM tool
    Calls tools.verifier.verify_statement(...); stashes the VerifyResult in deps.last_verify_result for the save-gate.
  • save_result(fields_json)LLM toolsave-gate
    See save gate. Writes {stmt}_result.json and {stmt}_cost_report.txt.

The save-gate — why agents can't just exit

save_result is refused unless all of these are true:

  • deps.last_verify_result is not None (verifier was run since the last fill)
  • last_verify_result.is_balanced is not False
  • mandatory_unfilled == [] (every *-prefixed mandatory row has a value or formula in the CY column[s])

Escape hatch — _is_force_save_allowed — kicks in when deps.turn_counter ≥ MAX_AGENT_ITERATIONS − 3 (i.e. iteration ≥ 37 of 40). At that point the agent is allowed to save partial work rather than the run dying silently.

Fuzzy label matching — tools/fill_workbook.py

Inside fill_workbook → _find_row_by_label:

  • auto Library: difflib.SequenceMatcher (stdlib — NOT rapidfuzz).
  • threshold Match accepted at ratio ≥ 0.7 for face statements. (Notes raise the bar to 0.85 — see §4.)
  • Built once per sheet via _build_label_index → list of _LabelEntry(normalized_label, row, section, is_header).
  • Leaf-preferred-over-header: when the same normalised label appears at both a header row and a leaf row (e.g. "Other fee and commission income" appears as a section header AND as a specific leaf in SOPL-Analysis), the leaf wins. This is the row-based fix from gotcha #17.

Double-booking guard — _detect_double_bookings

If two fills land the same (sheet, col, value) with ≥ 3 shared evidence tokens of ≥ 4 chars each, a warning is bubbled up. Column-scoped so legitimate Group consolidation pass-through (the same number appearing in both Group col B and Company col D) doesn't trip the guard.

Abstract-row guard — the most important refusal

XBRL-abstract rows (dark-navy section headers like "Non-current assets") must never carry a value — they're scaffolding, not facts. Two coupled defences live in the writer:

  1. guard Fill colour check: _HEADER_FILL_RGB = {"001F3864", "FF1F3864", "00305496", "FF305496"} (dark navy + mid-blue). Any row in col A with this fill rejects writes with an actionable error pointing the agent at the leaves below.
  2. guard Total-row colour check: _TOTAL_FILL_RGB = {"00EEF2F8", "FFEEF2F8"} rows are formula totals — also refused.
  3. Any cell whose existing value starts with = (a formula) is refused regardless of fill.

Prompt-level reinforcement lives in prompts/_base.md, prompts/sopl.md, and prompts/correction.md: "never plug a residual into a catch-all row to balance verify_totals". The verifier's SOFP imbalance feedback explicitly carries the line "do NOT plug a catch-all row", pinned by tests/test_verifier_feedback_wording.py.

MPERS parity The MPERS generator (scripts/generate_mpers_templates.py · _apply_abstract_row_styling) paints the same 1F3864 fill so the guard works identically on both standards.

Verifier — tools/verifier.py

The verifier is a custom Excel formula evaluator — it does not rely on openpyxl's formula engine (openpyxl can't evaluate). It walks the openpyxl.formula.tokenizer.Tokenizer token stream itself, with cycle detection via a visited set. Tolerance for balance comparison: 0.01.

StatementWhat it checks
SOFPAssets == Equity + Liabilities. On Group: also Group attribution = owners + NCI.
SOPLProfit (loss) == owners' share + NCI share.
SOCITotal CI = P&L + Total OCI, plus attribution split.
SOCFcash_end = cash_beginning + net_increase_after_fx; and operating + investing + financing = net_increase_before_fx.
SOCIEClosing equity = restated opening + total movement. Column resolution standard-aware: MPERS col 2, MFRS col 24 (matrix layout).

Supports cross-sheet references, SUM(range, …) via _expand_range, signed sums in coefficient form (=1*B7+-1*B11), and unary minus for negative cashflow signs. Unsupported tokens emit warnings and return 0.0.

_collect_unfilled_mandatory scans col A for *-prefixed labels; a row counts as unfilled only if every CY column (col B; or B+D on Group) is literally blank — formulas count as filled.

§4 Notes pipeline (sheets 10–14 MFRS / 11–15 MPERS)

Notes templates carry policy text, related-party disclosures, share capital tables and the list-of-notes. Five per filing standard, run in parallel with the face statements. Discovery is PDF-first: scout builds notes_inventory; per-template agents read those pages and write content. There is no deterministic synonym matching — pure LLM judgement.

Tools the notes LLM can call

  • read_template()LLM tool
    Returns col-A labels only (async via asyncio.to_thread). The col-A vocabulary is also seeded into the system prompt so MPERS agents pick from the live template wording rather than their MFRS training prior.
  • view_pdf_pages(pages)LLM tool
    Same shape as the face agent.
  • write_notes(payloads_json)LLM toolguards
    Each NotesPayload carries chosen_row_label, content (HTML), evidence, source_pages, numeric_values, note_num, source_note_refs, parent_note (optional), sub_note (optional). In Sheet-12 sub-agent mode this routes to _sub_agent_sink_write which collates rows; otherwise it writes directly via notes.writer.write_notes_workbook.
  • save_result(payloads_json)LLM tool
    Writes NOTES_{type}_result.json + cost report. No-op in sub-agent mode.
  • submit_batch_coverage(receipt_json)conditional
    Registered only on Sheet-12 sub-agents (when deps.batch_note_nums is set). Forces the sub-agent to acknowledge every note in its batch as either {"action":"written"} or {"action":"skipped"}.

Sheet-12 (LIST_OF_NOTES) — N-way sub-fan-out

The "list of notes" sheet has one row per disclosed note (and the inventory can run to dozens of notes). One agent would saturate context window. Solution: notes.listofnotes_subcoordinator.run_listofnotes_subcoordinator:

  • split_inventory_contiguous(inventory, n_batches) — page-contiguous slicing (not round-robin) so each sub-agent sees a coherent run of pages.
  • parallel Fan-out width = pricing.resolve_notes_parallel(model): DEFAULT_NOTES_PARALLEL = 5, drops to 2 for fast/mini models per config/models.json, capped at _MAX_NOTES_PARALLEL = 10.
  • Each sub-agent gets an extended iteration budget: max(MAX_AGENT_ITERATIONS, _BOOTSTRAP_ITERATIONS=10 + _PER_NOTE_ITERATIONS=4 × batch_size).
  • Catch-all: row 112 ("Disclosure of other notes to accounts") collects any unmatched note disclosures from sub-agents.
  • guard Empty-inventory fail-loud: if scout returned no notes (scanned PDF where even vision found nothing), Sheet-12 refuses to silently ship an untouched template. Carve-out: every sub-agent submitting a clean "skip all" receipt copies the template verbatim and reports success-with-warnings.

HTML sanitiser — notes/html_sanitize.py

Notes cells are HTML, not plain text. Sanitiser runs every payload before persisting.

Allowed tags:

ALLOWED_TAGS = {
  "p", "br", "strong", "em",
  "ul", "ol", "li",
  "table", "thead", "tbody", "tr", "th", "td",
  "h3"
}

Destroyed: script · style · iframe · object · embed (tag + contents). Unknown structural tags are unwrapped (contents kept). All on* handlers, style, class, id, src, href, action are stripped. Bare prose is auto-wrapped in <p>.

Heading injection: parent_note + optional sub_note are prepended to the cell as <h3> lines — owned by the writer, not the agent. But in-prose sub-section labels like "(a) Short term benefits" must be preserved verbatim as <p><strong>…</strong></p> — see gotcha #16.

Label normalisation (notes/labels.py · normalize_label): strips leading *, lowercases, strips one trailing taxonomy suffix from:

_TAXONOMY_SUFFIXES = (
  "[text block]", "[textblock]",
  "[abstract]", "[axis]", "[member]",
  "[table]", "[line items]"
)

DB persistence, overlay, and clobber-on-rerun

The on-disk xlsx is a flattened snapshot — the DB is the source of truth:

agent HTML → sanitiser → notes_cells (DB, canonical) → overlay → xlsx stream
                                       ↘ NotesReviewTab (TipTap editor)
  • persist_notes_cells writes inside a single WAL transaction; calls repo.delete_notes_cells_for_run_sheet first → even an empty payload wipes stale rows (clobber-on-rerun).
  • The Excel download endpoint calls overlay_notes_cells_into_workbook(xlsx_path, run_id, db_path) at stream-time — each DB row is flattened via html_to_excel_text and applied to col 2 of its sheet/row. Formula cells are refused.
  • The 30 K char cap is rendered length (via rendered_length), not raw HTML bytes. The PATCH editor endpoint returns 413 if edited content exceeds it.
  • Re-running an agent on a sheet that the user has edited triggers a confirm dialog — GET /api/runs/{id}/notes_cells/edited_count reports rows with updated_at > run.ended_at. Agent re-run still clobbers.
  • Evidence column is read-only in the editor; the PATCH endpoint ignores any evidence key in the body.

§5 Workbook merger — stitching it all together

Once every per-statement and per-notes-template workbook is on disk, workbook_merger.merge(...) copies each sheet into a single filled.xlsx.

  • Sheet order: StatementType enum order first (SOFP → SOPL → SOCI → SOCF → SOCIE), then NotesTemplateType enum order (CORP_INFO → ACC_POLICIES → LIST_OF_NOTES → ISSUED_CAPITAL → RELATED_PARTY).
  • All sheets within each per-statement workbook are copied — so SOFP carries both SOFP-CuNonCu and any sub-sheets/analysis sheets.
  • _copy_sheet preserves values + formulas verbatim + every style attribute (font, fill, border, alignment, number_format, protection), column widths, row heights, merged ranges. Formulas are not remapped because templates have no cross-statement refs.
  • Sets fullCalcOnLoad / forceFullCalc / calcOnSave = True so Excel recomputes the moment the file is opened.
  • Post-merge tools.recalc.recalc_workbook runs unless skip_recalc=True — skipped on the success path so the correction agent can run against unevaluated formulas first; final recalc happens after correction.
Stop-All preserves partial work
If the user cancels mid-run, server._attempt_partial_merge fires from the CancelledError branch before the audit row is marked aborted. Any per-statement {stmt}_filled.xlsx files already on disk are stitched into a partial filled.xlsx and surfaced via a partial_merge SSE event with included/missing statement lists. Pinned by tests/test_stop_all_preserves_partial.py.

§6 Cross-checks — does it actually add up?

Once merged, cross_checks.framework.run_all(...) walks each check. Each cross-check is a small class that declares which workbooks/sheets it needs, which standards/variants it applies to, and returns a CrossCheckResult(status ∈ {passed, failed, pending, not_applicable, warning}).

Filtering order before invoking a check

  1. Required statements missing? → pending.
  2. Required workbook file missing? → failed.
  3. Filing standard not in applies_to_standard? → not_applicable.
  4. applies_to(run_config) returns False (variant gate)? → not_applicable.
  5. Otherwise run it.

Default tolerance: DEFAULT_TOLERANCE_RM = 1.0. Default standard set: {"mfrs", "mpers"}. Each result fires an on_check(idx, total, result) callback in a finally: block — that callback is what feeds the live cross_check_result SSE events.

§7 Correction agent — fixing what failed

If the initial cross-check pass produces any status="failed" result, server._run_correction_pass spawns a single correction agent (correction/agent.py) that works directly against the merged filled.xlsx until checks pass or budgets run out.

Three independent budgets — defence in depth

  • turn cap max_turns = max(8, min(25, 8 + (4 if is_group else 0) + 2 × n_failed)) — RUN-REVIEW P0-1 dynamic 8–25 cap. Exhausted → SSE error.type = "correction_exhausted".
  • per-turn CORRECTION_TURN_TIMEOUT = 180.0 s wraps each __anext__.
  • wall-clock CORRECTION_WALLCLOCK_TIMEOUT = 300.0 s (env override XBRL_CORRECTION_WALLCLOCK_S). Catches slow-LLM scenarios where many quick-but-not-quick-enough turns add up. Exhausted → error.type = "correction_wallclock_exceeded".

Tools the correction agent can call

  • inspect_workbook(query_json)LLM tool
    Read-only lookups: by label or by row number, returns the target row plus context_rows surrounding rows (default 1, max 5) and nearby formulas that reference the target row (window 10, max 12 refs). Lets the agent see the full picture before patching.
  • view_pdf_pages(pages)LLM tool
    Same as face agent.
  • fill_workbook(fields_json)LLM toolguards
    Writes in-place to the merged workbook (template == output == merged file). All the standard guards apply — abstract-row refusal, double-booking warning, leaf-preferred-over-header.
  • verify_totals(statement)LLM tool
    Re-runs the verifier against the merged workbook for a single statement.
  • run_cross_checks()LLM tool
    Re-runs all cross-checks, scoped to deps.statements_to_run so partial runs don't fabricate missing-sheet failures.

§8 Notes validator — Sheet 11 ↔ Sheet 12 dedup

Sheet 11 (Summary of Accounting Policies) and Sheet 12 (List of Notes) often duplicate content — the same disclosure ends up in both with slightly different wording. The notes validator runs once after merge to detect and resolve this.

Detection is purely deterministic before the LLM enters the loop:

  • detect_cross_sheet_duplicates_by_ref — exact match via source_note_refs.
  • detect_cross_sheet_overlap_candidates — char-shingle Jaccard with _SHINGLE_SIZE = 5, _OVERLAP_THRESHOLD = 0.5.

Tools exposed to the validator agent:

  • view_pdf_pages(pages)LLM tool
  • read_cell(sheet, row, col)LLM tool
  • rewrite_cell(sheet, row, col, content, evidence?)LLM toolguards
    Allowed sheets are hard-restricted to _REWRITE_ALLOWED_SHEETS = {"Notes-SummaryofAccPol", "Notes-Listofnotes"}. Any formula cell refused. Empty content clears the cell + evidence column.
  • flag_duplication(note_ref, decision, rationale)LLM tool
    Appends to correction_log, persisted to notes_validator_log.json in the output directory.

Budgets: NOTES_VALIDATOR_TURN_TIMEOUT = 180.0 s, NOTES_VALIDATOR_WALLCLOCK_TIMEOUT = 300.0 s.

Iteration cap — why MAX_AGENT_ITERATIONS = 40 not 50

Gotcha #18 — caps must stay below pydantic-ai's silent 50
MAX_AGENT_ITERATIONS is resolved at import by _resolve_max_iterations(). Default 40, env override XBRL_MAX_AGENT_ITERATIONS, clamped to a _SAFE_CEILING = 45. The buffer of 5 absorbs pydantic-ai's per-iteration request overhead — if our cap reached 50 we'd race pydantic-ai's silent UsageLimits.request_limit = 50 and surface UsageLimitExceeded instead of our structured "Hit iteration limit" path. Pinned by tests/test_max_agent_iterations_below_pydantic_cap.py.

Per-turn timeout: FACE_TURN_TIMEOUT = NOTES_TURN_TIMEOUT = CORRECTION_TURN_TIMEOUT = 180.0 s. Implemented by _iter_with_turn_timeout wrapping iterator.__anext__() in asyncio.wait_for(...).

Trace persistence

save_agent_trace(result, output_dir, prefix) writes {output_dir}/{prefix}_conversation_trace.json, eliding any data/content field larger than _STRIP_THRESHOLD_BYTES = 500 via strip_binary. Write failures are logged but never raised — trace failures never mask the actual run result.

Token accounting

Gotcha #6 — per-turn token counts are approximate
_track_turn() records zeros for per-turn tokens because PydanticAI does counting internally. After completion, server.py backfills totals from agent_run.usage() into TokenReport. The UI's token dashboard shows real numbers only after the run finishes.
