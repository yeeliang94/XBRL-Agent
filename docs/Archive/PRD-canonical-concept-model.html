Canonical Concept Model for XBRL Extraction

PRD draft · v0.2 · 2026-05-21 · author: William + Claude · revised after peer review
v0.2 changelog (2026-05-21): Seven revisions after peer review. (1) Split single status field into two axes — value_status + children_status. (2) Fact records now carry period + entity_scope dimensions. (3) Phase 1 disables auto-correction on canonical runs; reconciliation queue moves into Phase 1. (4) Display label overrides confined to UI; Excel export always writes canonical SSM labels. (5) aggregate_only export now explicitly replaces the formula with a literal value + source-column annotation. (6) Phase 4 template count corrected from "14" to enumerated list (38 files / 20 functional combinations). (7) Concept ID separated into immutable UUID + render key + display + canonical labels; DB schema strawman added.
The change in one paragraph: Today the AI agent writes directly into .xlsx cells. Tomorrow it writes into a structured concept tree in the system, and Excel becomes a final export at the end. This solves three problems at once: users edit fields in a clean UI instead of Excel, the agent sees relationships explicitly instead of inferring them from spreadsheet structure, and conflicts between agents become visible and resolvable instead of hidden in a workbook.

1. Why we're doing this

Three pain points, all rooted in the same cause: Excel is currently both the source of truth AND the agent's working medium.

Pain pointSymptom todayRoot cause
Editing line items is fiddly Users have to open Excel, find the right row, type into the right column, save, re-upload. No system-side representation of the line items exists. The workbook IS the data.
Agents don't grok relationships Agent fills face sheet cell directly instead of sub-sheet (which face pulls from). Doesn't notice a sub-total mismatch until cross-check runs at the end. Excel hides the dependency graph inside opaque formulas. Agent prompt only shows column-A labels and cell coordinates.
Correction agent is finicky Plugs residuals into "Other" rows to make verify_totals pass. Exhausts its 8–25 turn cap reasoning about cell relationships. Sometimes fixes wrong cell because it can't see the cascade. Same as above — Excel doesn't surface "you wrote 50K at parent but children sum to 47K" until much later, and when it does, the fix has no clean target.

The underlying assets are XBRL concepts in a hierarchical taxonomy. Excel is a presentation/submission format — it shouldn't be the agent's mental model.

4. The concept tree

This is the new shape of the data. Every line item in a template becomes a concept node. Concept nodes have one of three kinds:

LEAF — agent fills this from the PDF
COMPUTED — system derives from children
ABSTRACT — header label only, no value
ABSTRACT Statement of Financial Position
ABSTRACT Assets
ABSTRACT Non-current assets
COMPUTED *Total property, plant and equipment = sum of 13 children
COMPUTED *Land = freehold + leasehold + other
LEAF Freehold land
LEAF Leasehold land
LEAF Other land
COMPUTED Total buildings = sum of 4 children
LEAF Office buildings
LEAF Factory buildings
...

Where this tree comes from

At template-import time, the system walks the .xlsx with openpyxl and tags every row:

Detected from templateBecomes
Dark-navy header fill (gotcha #17)ABSTRACT
Cell contains a formula like =1*B10+1*B11+...COMPUTED with parsed dependencies
Cell empty, not abstract, not computedLEAF
Cell contains ='Sub-sheet'!B39Edge in the dependency graph; not a separate concept (face is just a view of sub)
Why this is cheap: formula grammar in these templates is tiny — only three shapes (signed sums, single cross-sheet refs, occasional SUM() ranges in SOCIE). The parser is ~40 lines of code. Confirmed by the discovery pass on 5 representative templates.

The sub-sheet question, answered

Today, agents must know: "Write to SOFP-Sub-CuNonCu!B22, the face sheet will pull it." This is in the prompts because Excel forces it.

In the concept tree, the face/sub distinction disappears. The face sheet's ='SOFP-Sub-CuNonCu'!B39 formula isn't a separate concept — it's just a rendering edge. From the agent's perspective, there's only "PPE land freehold" (a leaf), with a known parent chain that gets recomputed automatically. The "fill sub before face" rule goes away.

5. Two status axes per fact

Every value carries two independent statuses: one describing the value itself, one describing how its children behave (for non-leaf concepts). The original PRD draft conflated these; the peer review correctly flagged that they're orthogonal concerns.

Axis 1 — value_status (applies to any concept that has a value)

StateMeaningWhen used
observed Value directly read from PDF Default for agent-extracted leaf values
explicit_zero PDF explicitly states zero / nil / "—" Distinguishes "PDF said zero" from "agent didn't fill"
not_disclosed PDF is silent on this concept Agent has searched, found nothing; leaf stays blank but documented
user_override Value entered or edited by a human via settings UI Survives agent re-runs unless explicitly cleared
conflict Two sources disagree (cross-statement, or agent vs user) Surfaced in reconciliation queue

Axis 2 — children_status (applies only to COMPUTED parent concepts)

StateMeaningWhat the validator doesWhen the agent uses it
itemised Children filled; parent = sum of children Verifies sum. Mismatch = error. PDF breaks down the line into its components.
aggregate_only Value stored at parent. Leaves intentionally blank. PDF did not itemise. Accepts. Does NOT complain about empty children. PDF shows "Trade and other receivables 50,000" with no breakdown.
partial Some children filled AND a parent value exists; residual. Flags as conflict. Goes to reconciliation queue. Error state — agent shouldn't pick this deliberately.
Why this matters for finickiness: Today the correction agent's only way to make a parent total tally is to invent values for empty leaves or plug into a catch-all "Other" row (gotcha #17). With aggregate_only as a first-class children_status, "the PDF doesn't break this down further" is a legitimate, documented answer. With not_disclosed as a first-class value_status, "the PDF doesn't mention this concept" is also documented rather than indistinguishable from "agent forgot to fill it." The structural incentive to fabricate disappears on both axes.

6. End-to-end walkthrough: SOFP CuNonCu Company

Let's trace one specific line item — Trade and other current receivables — through the full pipeline. PDF discloses 50,000 with no breakdown. Template has 9 leaf rows under this concept.

Step 1 — Template import (one-time)

COMPUTED *Trade and other current receivables [row 169, formula = R148 + R168]
COMPUTED Total current trade receivables [row 148, sum of R139..R147]
LEAF Trade receivables - third parties (rated/secured)
LEAF Trade receivables - third parties (rated/unsecured)
...7 more leaves...
COMPUTED Total other current receivables [row 168]

Step 2 — Agent reads PDF page 28

PDF reads: "Trade and other receivables    50,000". No further breakdown in the notes either.

Step 3 — Agent decides children_status

Agent's prompt context now includes the concept tree. It sees that "Trade and other current receivables" is a COMPUTED node with 9 fillable children. It also sees the aggregate_only option in its writer toolspec.

Agent emits a fact record. The composite key is (concept_id, period, entity_scope) — a single concept can carry up to four facts on a Group filing (CY/PY × Group/Company):

{
  "concept_uuid": "c0a8f5e2-4b1a-4e9d-9c3a-1f2d4b8e9a01",
  "render_key": {
    "template": "MFRS-SOFP-Company-v1",
    "sheet": "SOFP-Sub-CuNonCu",
    "row": 169
  },
  "display_label": "*Trade and other current receivables",
  "canonical_label": "*Trade and other current receivables",
  "period": "CY",                  // CY | PY
  "entity_scope": "Company",       // Company | Group (only meaningful on Group filings)
  "value": 50000,
  "value_status": "observed",      // observed | explicit_zero | not_disclosed | user_override | conflict
  "children_status": "aggregate_only",
  "evidence": {
    "pdf_pages": [28],
    "quote": "Trade and other receivables 50,000",
    "note_ref": null
  },
  "provenance": {
    "agent": "sofp_company",
    "turn": 7,
    "written_at": "2026-05-21T14:33:02Z"
  }
}

Currency (MYR) and unit (RM) are implicit by SSM convention and not stored per-fact.

Step 4 — Live validation

System checks:

  • Concept is computed-with-aggregate-permission? ✓ (any aggregator concept allows it)
  • Children all empty? ✓ (none have been written by other agents)
  • No conflict with parent's parent? ✓ (will be checked when parent is computed)

Write accepted. Concept marked aggregate_only. No "leaves don't sum to 50K" complaint.

Step 5 — Cascade re-evaluation

System recomputes everything that depends on this concept:

  • Total current assets (parent): includes 50K from this concept
  • Total assets (grandparent): includes the new current total
  • Balance check vs equity + liabilities: re-run automatically

Step 6 — Merge with other agents

SOPL agent has been running in parallel and produced a profit for the year of 200,000. SOCIE agent has produced a closing equity of 1,150,000. Cross-checks now run on the canonical model:

  • Profit → SOCIE retained earnings movement: ✓
  • SOCIE closing equity → SOFP equity: ✓
  • Assets = Equity + Liabilities: ✓

Step 7 — Export to Excel

User clicks Download. System opens the live template, walks every leaf concept and stamps its value into the matching cell. The 9 trade-receivables leaves stay empty in Excel (that's correct — they're aggregate_only). Excel's own formulas recompute totals on open. A small footnote in cell D169 records the aggregate_only status for the auditor.

7. How this changes the correction agent

You called this out specifically. Here's the before/after.

Today

Input: "Cross-check failed: assets 1,200K but equity+liabilities 1,150K. 50K imbalance."

What the correction agent sees: A list of cells with values. No relationships exposed. It must reason about which leaf, on which sheet, contributes to which parent.

Failure mode: Tries 15 turns. Hits dynamic cap. Plugs 50K into Other current assets to balance — wrong, but verify_totals passes. Wallclock cap (300s) sometimes fires before resolution.

Result: Run lands completed_with_errors or with hidden fabrication.

Tomorrow

Input: "Cross-check failed: Total assets concept = 1,200K (sum of 14 children, all itemised). Total equity+liabilities concept = 1,150K (sum of 18 children, one is aggregate_only at 50K). 50K imbalance."

What the correction agent sees: Full concept tree with values, provenance, evidence pages, and children_status for every node.

Resolution options it has:

  1. Re-read PDF pages for one of the aggregate_only concepts; maybe a breakdown exists in a note it missed
  2. Mark a parent as aggregate_only on either side if the imbalance is a known PDF disclosure choice
  3. Push to the reconciliation queue if it can't decide

Crucially: "plug into Other" is no longer the path of least resistance.

What stays the same

  • Dynamic 8–25 turn cap (RUN-REVIEW P0-1) — still applies
  • 300s wallclock cap (gotcha #18) — still applies as defence-in-depth
  • Below-50 iteration cap relative to pydantic-ai's silent limit (gotcha #18) — still applies

What gets removed

  • The Excel-cell mental model in the correction prompt
  • The "don't plug residuals" prompt fragment (the affordance no longer exists; aggregate_only is the legitimate alternative)
  • The face/sub-sheet ordering rule
Rollout note (v0.2): The migration above describes the end-state after Phase 3. In Phases 1–2, auto-correction is disabled on canonical-mode runs. Cross-checks still run and surface failures to the reconciliation queue for human resolution. The reason: a Phase-1 correction agent that wrote into Excel would desync the DB canonical values; a correction agent that wrote into the DB would need its prompts rewritten and tests re-pinned, which is Phase 3's whole scope. Disabling correction in the interim is cleaner than maintaining two parallel correction flows.

What users can do

  • Edit values on any LEAF concept
  • Rename display labels on any concept (canonical SSM label preserved underneath, shown in tooltip)
  • See computed values with their dependency formula
  • Export to Excel at any time
  • Re-import template when SSM regenerates (handled separately — see open decisions)

What users cannot do (intentionally)

  • Add / remove rows (that's a template-version change, not a per-run edit)
  • Override formulas (those come from SSM)
  • Edit COMPUTED concept values directly (they're derived)
  • Edit ABSTRACT concepts (they have no value)

Non-technical user concern (you raised this)

Most SSM labels read fine after stripping the * prefix and [abstract] / [member] suffixes. The ~5% genuinely ugly ones get a display_label_override:

SSM canonical (hidden, in tooltip)What user sees by defaultAfter user renames
*Total non-current portion of non-current secured bonds/sukuk/loan stocks Total non-current portion of non-current secured bonds/sukuk/loan stocks Long-term secured bonds (LT portion)
[abstract] StatementOfFinancialPositionAbstract Statement of Financial Position (no override needed)

The canonical label is always one click away (the ℹ tooltip) so auditors can verify what XBRL concept any line maps to.

Important boundary (peer-review fix): display_label_override is a UI-only field. It NEVER appears in the exported Excel file. The exported .xlsx always carries the canonical SSM label in column A, because M-Tool (SSM's submission validator) keys ingestion off canonical taxonomy labels. If a user renames "Receivables" to "Customer debts" in the UI, the Excel still says "Receivables" — the rename only changes what they see in the /concepts page and the reconciliation queue. This is non-negotiable; the alternative is silent submission failures we couldn't debug.

10. Excel export — the final deliverable

Excel remains the submission format. The export path is mostly straightforward but has one subtle case (aggregate_only) that needs explicit handling.

Standard path (itemised concepts)

  1. Open the live template .xlsx in openpyxl (read-write mode)
  2. For every LEAF fact in the run with value_status ∈ {observed, explicit_zero, user_override}, stamp the value into its render-key target cell
  3. Skip leaves marked not_disclosed — they stay blank, and a side-channel record carries the documentation
  4. Save; COMPUTED cells' formulas recompute on open

Special path (aggregate_only concepts) — the peer-review fix

The conflict that v0.1 missed: the parent cells where I proposed stamping aggregate values are themselves formula cells (e.g. SOFP sub R169 = =R148+R168). The current writer at tools/fill_workbook.py:291 explicitly refuses to overwrite formulas. So "stamp at parent + keep formulas live" was internally contradictory.

Resolution: for any concept with children_status = aggregate_only, the export does this:

  1. Replace the formula in the parent cell with the literal aggregate value. The formula =R148+R168 becomes the number 50000.
  2. Write an annotation in the Source column of the same row: "aggregate_only — pdf p.28; underlying breakdown not disclosed"
  3. Leave all child leaf cells blank
  4. Record in concept_fact_events that the formula was replaced, so the source-of-truth audit trail is intact

This means the "Excel formulas stay live" promise is qualified — it applies to itemised concepts only. Aggregate concepts ship as static values. That's the correct behaviour: the agent has explicitly said "I cannot itemise this," so there is no live sum to compute.

Label handling

Excel export always writes the canonical SSM label into column A — never the user's display_label_override. M-Tool (SSM's submission validator) keys ingestion off canonical taxonomy labels; mutating column A could silently fail submission. The user's override is purely a UI affordance for the /concepts page and never leaves the system.

~75% the same code as today's tools/fill_workbook.py, reading from the DB instead of from agent writes. The header guard (tools/fill_workbook.py:291) gets a new branch for the aggregate_only override path.

12. What this does NOT solve

Being honest about the limits.

  • Agents still misread numbers from PDFs. The canonical model makes misreads more visible, but doesn't prevent them.
  • Some PDFs are internally inconsistent. The reconciliation queue gives the user a structured way to handle this, but the underlying inconsistency remains.
  • Concept ID stability across SSM template updates — parked on the back burner per your direction. Re-import with row-shift handling is needed eventually.
  • SOCIE's 24-column matrix doesn't fit the linear leaf/computed/abstract model cleanly. Needs its own schema variant.
  • The user's edits to a draft are not automatically synced into a running extraction. Live editing during a run is out of scope.

13. Decisions (locked in)

All seven open decisions are now resolved. Plain-language version of each:

#DecisionWhat it means in plain language
1 Concept identity split into four fields (revised v0.2 after peer review) v0.1 baked row + label slug into a single ID, but those change on template re-import. Revised structure:
  • concept_uuid — immutable UUID, DB primary key. Never changes.
  • render_key — {template, sheet, row} tuple. Where to stamp the value in Excel. Re-mapped on template re-import.
  • canonical_label — SSM source-of-truth label text. Read-only.
  • display_label — what the user sees in the /concepts page. May be overridden via UI. Never written to Excel (per decision in section 9).
Foreign keys in DB always use concept_uuid. Render keys live in a separate concept_targets table.
2 Database: extend the existing local SQLite DB (schema v3 → v4) Same file as today (runs, notes_cells already live there). New tables (strawman):
  • concept_templates — one row per imported template version
  • concept_nodes — concept_uuid, parent_uuid, kind (LEAF/COMPUTED/ABSTRACT), canonical_label, display_label_override
  • concept_edges — formula dependencies: parent_uuid, child_uuid, coefficient (±1)
  • concept_targets — render addresses: concept_uuid, sheet, row, col, period, entity_scope
  • run_concept_facts — per-run values, keyed by (run_id, concept_uuid, period, entity_scope); carries value, value_status, children_status, evidence_json
  • run_concept_conflicts — reconciliation queue entries: run_id, conflict_type, payload_json, resolution_status
  • concept_fact_events — append-only event log of all writes for provenance (agent, turn, timestamp, before_value, after_value)
Migration is idempotent and gated by version probe, same pattern as v1 → v2 (gotcha #11).
3 Agents write via a coordinator-owned API, not direct DB Agent emits a structured write to the coordinator (same pattern as today's notes flow). Coordinator validates against the concept tree, applies the three-state rule, and either accepts or rejects with a reason. This is where the "you wrote at parent AND child" detection lives — keeping it in one place stops every agent from re-implementing it.
4 SOCIE: defer to Phase 4 SOCIE has a 24-column matrix structure that doesn't fit the linear leaf/computed/abstract model. Phases 1–3 stick to the simpler face/sub-sheet pattern. When SOCIE arrives, it gets its own schema variant (concept = row × column instead of just row). Same DB, additional table.
5 Live re-evaluation: batch at agent-turn boundary An agent typically writes multiple values in one turn (e.g. fills 8 leaves of a section). Rather than recomputing the dependency graph after each individual write, recompute once at the end of the turn. Less DB churn, same effective real-time feel, simpler to reason about.
6 UI: new /concepts page New top-level page. Existing draft / history / notes-review flows link into it. Keeps the existing tabs uncluttered. The reconciliation queue lives on the same page as a side panel.
7 Legacy runs: kept, badged read-only Old runs (Excel-only) appear in history with a "legacy" badge. Download still works (uses the existing filled.xlsx on disk). Editing requires re-running through the new pipeline.

What I deliberately did NOT decide

  • Concept-ID stability across SSM template re-imports — parked per your direction. We'll need an answer before Phase 1 ships to real users, but Phase 0 doesn't depend on it.
  • Whether non-LEAF concept labels are renameable — leaning yes (parent labels are often the ones non-technical users care about), but will confirm in Phase 1 design.
  • Conflict resolution between two users editing the same run — single-user assumption holds for MVP. If multi-user editing becomes a need, we'll revisit.
