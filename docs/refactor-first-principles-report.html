Executive Summary

If by "AI slot" you mean the software now feels like AI slop or a slot-machine around an LLM, the root cause is not that it uses agents. The root cause is that too much product-critical state still flows through workbook side effects, long orchestrator functions, duplicate agent loops, and post-hoc reconciliation.

I would not rewrite the whole system in one shot. The domain assets are valuable: SSM templates, formula constraints, concept parser, fact store, notes HTML store, cross-checks, test suite, and Windows/provider invariants. The right move is a strangler refactor: make the canonical concept model the only authoritative extraction target, isolate LLM interaction behind a typed "evidence to fact" interface, and split the current server/orchestrator into small bounded services.

The desired end state is: PDF evidence in, typed facts out, deterministic validation and export after that. The LLM should judge evidence and disclose uncertainty; it should not be responsible for workbook mechanics, retry policy, lifecycle status, audit persistence, or cross-statement accounting consistency.

What the product is actually doing

The application converts Malaysian financial statement PDFs into SSM MBRS XBRL Excel templates. The business problem is not "fill cells"; it is "produce a defensible, auditable filing artifact from a messy source document."

EvidencePDF pages, TOC, statement pages, notes inventory, visual renders.
InterpretationLLM extracts values, prose, periods, entity scope, and evidence.
CanonicalizationFacts mapped to SSM concepts, periods, scopes, statuses.
ValidationFormula cascades, cross-checks, reviewer flags, conflicts.
ExportXLSX templates, notes overlays, downloads, audit history.

Important existing strengths

  • Canonical concept model exists. The project already has concept_model/parser.py, facts_api.py, exporter.py, concept targets, conflict rows, aliases, and versioning.
  • Domain invariants are documented. AGENTS.md and CLAUDE.md encode real failures: provider setup, Gemini temperature, pydantic-ai API changes, Windows UTF-8, template formulas, notes rules, lifecycle status, and canonical mode.
  • Tests pin behavior. The test suite covers schema migrations, canonical export, notes HTML, scout, provider routing, run lifecycle, SSE, cross-checks, MPERS, Group, SOCIE matrix, monolith, and reviewer flows.
  • Notes DB design is sound. Notes HTML in notes_cells as canonical source and XLSX as render target is the right pattern.
  • Cross-check framework captures accounting semantics. Those checks should become first-class validation rules, not afterthoughts.

Important existing liabilities

  • One file is doing too much. server.py is API layer, environment loader, model factory, SSE broker, coordinator, correction runner, download exporter, PDF endpoint, and history controller.
  • There are multiple orchestration loops. Split face agents, notes agents, notes sub-agents, correction, reviewer, validator, scout, and monolith each have overlapping timeout, event, usage, trace, and failure logic.
  • Workbook writes still leak into the core loop. Canonical mode projects from resolved workbook writes into facts. That is transitional, not first-principles.
  • The UI carries large local state reducers. web/src/lib/appReducer.ts, PreRunPanel.tsx, ConceptsPage.tsx, and review components are large enough that behavior is harder to reason about.
  • Model/provider behavior is embedded in runtime paths. Provider routing is centralized, but model capability, temperature, parallelism, and proxy limits are still scattered across config, docs, prompts, and callers.

Diagnosis

The core architectural smell is side-effect orchestration: agents act on XLSX files, server code observes many event streams, persistence happens during streaming, and later phases infer truth from artifacts. That creates a large surface for stuck runs, stale downloads, split-brain between DB and workbook, and "AI did something plausible" failures.

Where complexity is essential

Some complexity is real and should not be "simplified away":

  • SSM template formulas and linkbase-derived structure.
  • MFRS versus MPERS filing standards.
  • Company versus Group entity scope, including different column geometry.
  • SOCIE matrix geometry and MPERS SoRE behavior.
  • Notes pipeline as pure LLM judgement, with no deterministic label dictionaries.
  • Provider quirks, especially Windows enterprise proxy, Gemini temperature, and pydantic-ai version behavior.
  • Auditability: traces, per-turn telemetry, event history, terminal run statuses.

First Principles

A clean design starts by naming the invariants that should survive any implementation.

1. Evidence is immutable

Uploaded PDF, rendered pages, PyMuPDF text, scout output, and template versions should be content-addressed or snapshotted per run. Every fact must point back to evidence.

2. Facts are authoritative

The canonical DB should be the only source of truth for extracted face-statement values and notes HTML. XLSX is an export, not working memory.

3. LLMs propose, validators dispose

LLMs should propose fact writes with evidence and uncertainty. Deterministic validators decide formula consistency, dimensions, writable rows, periods, scopes, and exportability.

4. Every run is a state machine

A run moves through known states: draft, preparing, extracting, validating, reviewing, exporting, completed, completed_with_errors, failed, aborted. No path should infer status.

5. Domain model beats prompt text

Concepts, templates, dimensions, statuses, cross-checks, and formulas should be code/data. Prompts should render from that model instead of carrying duplicated accounting rules.

6. Provider differences are policy

Model routing, temperature, parallelism, retries, cache behavior, and token budgets should be a model policy layer, not embedded in each agent loop.

Unify the agent runtime loop

Face, notes, notes sub-agents, scout, correction, reviewer, validator, and monolith all repeat the same concerns: model call, iteration cap, per-turn timeout, usage deltas, tool events, trace saving, cancellation, and structured exhaustion. Build one runner. 

class AgentRuntime:
    async def run(
        self,
        spec: AgentSpec,
        deps: object,
        events: EventSink,
        limits: AgentLimits,
    ) -> AgentRunOutcome:
        ...

Each agent supplies only a prompt renderer, tools, deps, and output parser. The runtime supplies telemetry and safety. This will remove a surprising amount of bespoke code.

Change agent output from workbook writes to fact proposals

Today, face extraction still centers on fill_workbook. In the target design, a face agent proposes facts:

  "period": "CY",

  "children_status": null,

    "confidence": 0.82

The facts API already contains much of the validation surface. Promote it from route helper to application service and call it from every extraction path.

Rewrite Plan

A true rewrite should be measured in vertical slices, not modules. Each slice must preserve an executable product path.

Slice 1: Domain package and adapters

Create a new package beside the existing code. Move enums and value objects first: filing standard, filing level, statements, variants, concept kind, fact status, run status. Add zero behavior changes.

Exit: existing tests pass with compatibility imports.

Slice 2: Run lifecycle service

Extract run creation, draft start, terminal status, active lock, abort, and failure handling into RunService. Keep the same DB schema and endpoints.

Exit: lifecycle tests still prove failed/disconnected/cancelled runs never remain running.

Slice 3: Event bus

Replace ad hoc queue event dicts with typed event classes. Provide two subscribers: SSE stream and audit recorder.

Exit: live UI and history replay render the same event projections.

Slice 4: Fact-first face extraction

Introduce a new face extraction tool, propose_facts, that resolves labels/concepts before persistence. Keep legacy workbook write path behind a fallback flag.

Exit: SOFP Company canonical E2E passes without writing a scratch workbook.

Slice 5: Export-only workbook rendering

Make all face-statement downloads rebuild from facts. Remove scratch workbook dependency from merge for canonical-mode runs.

Exit: downloads, recheck, manual edits, and review UI all agree because they read the same facts.

Slice 6: Unified agent runtime

Port face, notes, scout, reviewer, validator, and monolith loops onto one runtime. Do this after facts are stable so there are fewer moving parts.

Exit: timeout, iteration cap, trace, usage, and cancellation code exists once.

Slice 7: UI projection API

Add a run projection endpoint that returns the current run page model: agents, events, stages, facts, conflicts, notes cells, telemetry, downloads. The frontend stops rebuilding so much meaning from raw events.

Exit: live and history screens share one data shape.

Data Model

The current DB is moving in the right direction. The next simplification is conceptual: one canonical write path for face facts and notes cells, with XLSX export as a deterministic projection.

Recommended fact lifecycle

  1. Proposed: An agent produces a fact proposal with evidence and confidence.
  2. Accepted: The validation service accepts it into run_concept_facts.
  3. Derived: Cascade recompute creates or updates computed facts.
  4. Conflicted: Parent/child or cross-check residuals create conflict rows.
  5. Resolved: Human or reviewer agent marks aggregate-only, not-disclosed, user override, or corrected observed value.
  6. Exported: Export service snapshots the final XLSX artifact.

Schema direction

  • Keep concept_nodes, concept_targets, concept_render_aliases, and run_concept_facts.
  • Keep notes_cells, but expose it through the same "fact proposal" mental model for orchestration.
  • Add an explicit fact_proposals or agent_observations table only if you need to preserve rejected writes for audit. Otherwise, keep rejected writes in trace/event history.
  • Keep schema migrations additive until a dedicated migration tool is adopted. Current SQLite walk-forward pattern is tedious but safe.
  • Consider splitting migrations out of db/schema.py after v12. A single schema file is becoming an archaeological layer.

LLM Strategy

The cleanest mental model is "small number of specialized LLM tasks, all surrounded by typed deterministic boundaries."

Scout

Finds structure: TOC, pages, variants, notes inventory, standard, periods, currency, entity. It should output only typed context, never extraction facts.

Extractor

Reads evidence and proposes facts. It should not write Excel. It should not decide whether a formula total can be overwritten.

Reviewer

Works only on conflicts and flags. It sees the fact graph, evidence, and validation residuals, then proposes resolutions.

Split versus monolith

The monolith experiment asks a good question: does shared context improve first-pass cross-statement consistency? I would keep it as an experiment, but not make it the main architecture yet. The main architecture should make cross-statement state shared through the fact graph and validation engine, not by putting all responsibility into one giant prompt.

Approach Use it for Risk
Split specialist agents Parallel extraction with small context windows and simpler prompts. Weak first-pass cross-statement consistency unless fact graph feedback is strong.
Monolith agent Experimental benchmark and maybe complex filings where cross-statement coherence dominates. Large prompt, provider limits, harder debugging, possible wrong-but-consistent outputs.
Hybrid fact-graph loop Recommended target: specialists extract, shared validator returns conflicts, reviewer resolves. Requires good typed proposal/validation APIs.

Prompt policy

  • Prompts should be generated from the concept model, not hand-maintained per rule where possible.
  • Every prompt should describe allowed statuses: observed, explicit_zero, not_disclosed, aggregate_only, user_override.
  • Every agent should have a typed output contract and a rejection feedback loop.
  • Provider model policy should own temperature, retry, request limits, notes parallelism, and body-size limits.

Recommendation

Do a refactor, not a clean-room rewrite. The best new architecture is already half-born inside the current one: canonical concepts, DB-backed notes, formula/cross-check validation, and a serious regression suite. The job is to make that architecture explicit and remove the old workbook-side-effect control plane.

Decision

Make canonical facts the center. Treat Excel as a renderer. Treat LLMs as evidence interpreters. Treat server orchestration as an application service with explicit states and typed events.

  1. Extract model factory and run lifecycle service from server.py.
  2. Introduce typed event classes with compatibility to current SSE payloads.
  3. Move export/download/remerge logic into an ExportService.
  4. Promote facts API internals into a FactService usable outside FastAPI.
  5. Build a single SOFP Company fact-first extraction path in shadow mode.

First 90 days

  2. Port Group, SOCIE matrix, and MPERS only after Company is stable.
  3. Unify agent runtime loops once typed events and fact writes are stable.
  4. Add projection APIs for live/history/review screens.
  5. Run the monolith experiment as a benchmark, not as the architectural foundation.

What I would not do

  • I would not delete the existing split pipeline until fact-first canonical mode has real-PDF proof.
  • I would not simplify notes matching with deterministic dictionaries, because the project explicitly forbids that and tests pin the philosophy.
  • I would not hand-edit SSM template formulas.
  • I would not convert the frontend back to Tailwind/className while Windows support is a known constraint.
  • I would not make the monolith agent the default based on intuition. Measure it with the existing experiment plan.
