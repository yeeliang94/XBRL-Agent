Engineering Design Report

Rewriting the AI Processing Pipeline from First Principles

A diagnosis of the current PDF → XBRL extraction pipeline, the irreducible problem it actually solves, and a target architecture derived from that problem rather than accreted around it.

Scope coordinator · extraction · notes · scout · concept model · reviewer · server orchestration Method evidence-based code audit (3 parallel passes) Date 2026-05-29
How to read this

This is not a list of nits. It is an argument that the pipeline grew by accretion — each feature bolted onto the last — and that several of its hardest-to-maintain parts exist because the shape of the system never matched the shape of the problem. First-principles means: throw out what's there, write down what the task irreducibly requires, and only then ask what code that implies. The diagnosis is grounded in real file:line evidence; the proposal is opinionated on purpose.

01The irreducible problem

Strip away every feature and the system does exactly one thing:

Given a financial-statement PDF, produce a set of (concept, period, scope) → value facts that are grounded in the PDF and internally consistent, then render those facts into SSM's fixed XBRL templates.

Everything else — agents, tools, scout, notes, the web UI — is machinery for producing and validating that fact set. This framing matters because it tells us what the single source of truth must be: a typed table of facts with provenance. The current codebase half-discovered this (the canonical run_concept_facts store is exactly the right idea) but never committed to it, so it carries the fact store and three older representations of the same data simultaneously.

From the irreducible problem, four invariants fall out that any correct implementation must hold. Worth naming, because the current code enforces each one in an ad-hoc, scattered way:

InvariantWhy it's irreducible
Facts are the source of truthThe XLSX, the conversation trace, and the cross-check results are all derived. If two representations disagree, you have a bug by construction.
Every fact carries provenance"Grounded in the PDF" is meaningless without a page + evidence string attached to each value. This is also the entire audit story.
Consistency is computed, not assertedTotals roll up from leaves via the SSM linkbase — a total is recomputed, not extracted, unless the breakdown is genuinely undisclosed, in which case the agent marks the parent aggregate_only and its disclosed literal stands (the cascade stops at that boundary; cascade.py:7, facts_api.py:316, exporter.py:193). The cascade engine owns both behaviours and should be the only way a computed total gets a value.
The LLM proposes; deterministic code disposesThe model's job is judgement (which line is "Trade receivables"?). Validation, rollup, routing, and persistence are deterministic and must not live in prompts.

02The minimal machine

If you sat down today with the problem statement and nothing else, you would build five components and no more:

PDF ─▶ ┌──────────┐ page map + context ┌──────────────┐ │ SCOUT │ ───────────────────────▶ │ EXTRACTORS │ one per statement / note └──────────┘ (advisory hints) │ (LLM loop) │ proposes facts w/ evidence └──────┬───────┘ │ FactWrite(concept, period, scope, value, evidence) ▼ ┌────────────────────────┐ │ FACT STORE (truth) │ typed table + cascade engine │ validate · roll up │ rejects ungrounded / abstract └──────────┬──────────────┘ │ ┌──────────────────────┼───────────────────────┐ ▼ ▼ ▼ ┌──────────┐ ┌───────────────┐ ┌─────────────┐ │ CHECKER │ │ REVIEWER │ │ RENDERER │ │ consist- │─flags──▶ │ investigates │─fix──▶ │ facts ─▶ xlsx│ │ ency │ │ & repairs │ └─────────────┘ └──────────┘ └───────────────┘

Five components, one data structure flowing between them (FactWrite), one store. Note what is absent: a "merger" (facts are already merged in the store), a "legacy xlsx path" (the renderer is the only writer), an "exporter vs. filler" distinction (one renderer), and a separate notes pipeline (notes go through the same fact-write interface, routed to their own backend — see §4.4). The current system has all of these as separate, duplicated subsystems.

The good news

Three of these five already exist and are well-built: the fact store + cascade (concept_model/facts_api.py, cascade.py), the reviewer with its deterministic no-plug guard (correction/reviewer_agent.py:310), and the cross-check framework. The rewrite is mostly deletion and consolidation, not green-field construction. You are removing the three things that compete with the fact store, not inventing a sixth thing.

03Where the code diverged from the minimal machine

The audit surfaced the same root cause from three independent directions. I'll group the findings by the first principle each one violates, rather than by file.

3.1 — There are four representations of "a fact", not one

The minimal machine has exactly one place a value lives: the fact store. The current system has the same value living in up to four places, written by different code, kept in sync by hope:

RepresentationWritten byProblem
Agent's scratch filled.xlsxtools/fill_workbook.pyThe thing the LLM thinks it's editing
run_concept_facts (DB)cell_resolver.project_writesThe actual source of truth — but written as a side effect of the xlsx write, behind an exception swallow (extraction/agent.py:192)
Re-rendered canonical xlsxconcept_model/exporter.pyOverwrites the agent's xlsx at download time — so the agent's view and the user's download differ
On-disk {stmt}_conversation_trace.jsonagent_tracing.pyA fourth copy of the same numbers, orphaned if output_dir is cleaned

The most damning symptom: the canonical fact projection — the write to the real source of truth — is best-effort and swallowed:

# extraction/agent.py:188 — the write to the source of truth cannot fail loudly
except Exception:
    logger.exception("canonical fact projection failed for %s", deps.statement_type.value)
    return "Canonical projection error: facts may be incomplete (see logs)."

If this fails halfway, the run completes "successfully" with a half-populated fact store, a complete-looking xlsx, and no structured error. The fact store is treated as a derived cache of the xlsx, when the truth is the exact opposite.

3.2 — Control flow lives in prompt strings and tool error messages

The fourth invariant — LLM proposes, code disposes — is inverted in several places where deterministic state machines are encoded as English that the model must read and obey across turns.

As built — the save gate

The "honest completion" gate is a 4-step state machine implemented as tool-refusal text (extraction/agent.py:271–315). The model must: call verify_totals → get refused → call it again → get told to fix-or-acknowledge → call save_result(acknowledge_unresolved=true, reason=…). Try to acknowledge too early and it's refused with "First contact with the gap — refuse once."

State lives in ~6 mutable flags on ExtractionDeps (seen_unresolved_refusal, unresolved_reason, …). The transitions are invisible to type-checking and only discoverable by reading prose.

First principles — explicit state

The gate is a pure function of typed state, not a conversation:

enum SaveGate:
  UNVERIFIED → "run verify first"
  FAILED & ¬ack → "fix or ack"
  FAILED & ack & reason → OPEN
  PASSED → OPEN

The LLM still decides whether to acknowledge; the gate is deterministic and unit-testable in isolation, with zero mutable flags threaded through deps.

The same anti-pattern recurs in the fill tool's two modes (label-matching vs. explicit coordinates) selected only by a docstring (extraction/agent.py:610), and in the evidence-column routing the tool silently overrides regardless of what the agent asked for (fill_workbook.py:353).

3.3 — The agent contract is a stringly-typed JSON blob

Tools receive fields_json: str and do json.loads() behind a bare except (fill_workbook.py:219). pydantic-ai will validate a typed tool signature for free and inject its schema into the prompt — but the code opts out of this, so the model guesses the schema, emits malformed JSON, and learns about it only from a return-string round-trip. This wastes tokens and turns, and it is the direct cause of much of the defensive parsing code.

3.4 — Two of everything: dual pipelines, dual coordinators, triple correction

This is the single largest source of bulk and bug-surface. The system maintains parallel implementations of the same job because new approaches were added beside the old ones rather than replacing them:

Duplicated axisVariants live inApprox. cost
Extraction coordinatorcoordinator.py (990) + monolith/coordinator.py (1259)~400 lines of identical timeout / iteration-cap / telemetry / SSE logic, incl. a duplicated _IterationLimitReached that admits it is a "mirror"
Face vs. notes coordinatorcoordinator.py + notes/coordinator.py~2,000 lines; same retry/backoff/timeout/token patterns. Bug fixes do not propagate between them (the task-registry cleanup had to be re-added by hand to one side)
Correction logiccorrection/agent.py (484, legacy) + canonical_agent.py (637, autonomous pass superseded by the reviewer, but its _load_open_conflicts helper is still imported live) + reviewer_agent.py (895, live)3 system prompts, 3 turn-cap algorithms, 3 wrappers around the same apply_fact()
Canonical vs. legacy xlsx pathbranch points throughout server.py~3,600 lines gated on XBRL_CANONICAL_MODE / _facts_enabled / _reviewer_enabled — every test runs twice, every feature lands twice
Root cause

None of these dualities is load-bearing. The monolith was an experiment (it loses, predictably: one agent trying to be both greedy extractor and investigative corrector is good at neither). The legacy xlsx path is what the canonical path replaced. The canonical correction agent's autonomous pass is what the reviewer replaced. All three are still in the tree — and not cleanly severable: the legacy path and monolith are flag/branch-gated, while canonical_agent.py still leaks a helper into the live reviewer path. The minimal machine has one coordinator, one correction pass, one rendering path.

3.5 — server.py is a 6,853-line god file

A single async generator, run_multi_agent_stream, is 2,006 lines and inlines validation, extraction launch, merge, canonical export, two cross-check passes, correction/reviewer dispatch, notes validation, and persistence — with 15+ try/except blocks competing for the same scope. Reordering two phases means editing this function. Forty-plus FastAPI routes share the file.

Symptoms of the size: the SSE event vocabulary has sprawled into ad-hoc shapes (pipeline_stage, cross_check_start/result/complete/exception, partial_merge, merge_failed, generic error) with no shared schema, so the frontend pattern-matches on string types. And background re-review is tracked in an in-process dict (_REVIEW_TASKS, server.py:5447) that vanishes on restart, silently abandoning any in-flight pass.

3.6 — Failures are swallowed, not surfaced

40+ except Exception handlers with noqa: BLE001 and a "never let X break Y" comment. The intent (don't let telemetry crash a run) is correct; the execution conflates advisory failures with structural ones. A re-check that fails returns None and the endpoint serves HTTP 200 (server.py:2316); the SSE event queue drops events when full and only logs it (server.py:3556); a persistence failure in the notes path completes the run "successfully" while leaving the DB stale and skipping the post-validator (notes/coordinator.py:389).

3.7 — Determinism that hides from its own callers

The scout claims a "PDF-first, no-LLM" inventory, but on a scanned PDF it silently delegates to a vision LLM (scout/notes_discoverer_vision.py) with no flag in the Infopack recording which path ran. Downstream agents trusting the inventory as deterministic ground truth are, on those runs, trusting an unmarked LLM guess. Conversely, fuzzy label matching at a 0.85 SequenceMatcher threshold (notes/writer.py:55) silently routes prose into the wrong row — the comment itself admits the band below was "almost entirely wrong-label cases," yet matches at threshold log at DEBUG.

04Target architecture

The rewrite collapses the system onto the minimal machine. One truth store, one path through it, typed contracts at every boundary, deterministic plumbing extracted out of prompts.

4.1 — One fact store, written first, rendered last

Principle

The fact store is written directly and transactionally by extraction — not as a side effect of an xlsx write. The xlsx does not exist until render time, when it is produced from facts. There is no agent scratch workbook to drift from, no "export overwrites the agent's file" surprise, and the projection write can no longer be a swallowed best-effort because it is the write.

This single change deletes an entire class of bug (the four-representations problem) and removes fill_workbook.py's dual-mode complexity, the exporter's "fall back to the agent workbook when zero facts applied" branch, and the merge step entirely.

4.2 — One generic agent loop, parameterised

Face extraction, notes extraction, monolith, and the reviewer are the same loop: build an agent, stream agent.iter() with a per-turn timeout, enforce an iteration cap below pydantic-ai's 50, record telemetry deltas, emit SSE, handle the four terminal exceptions, save a trace. Extract this once:

# agent_runner.py — the ONE place the loop lives
async def run_agent(spec: AgentSpec, sink: EventSink) -> AgentOutcome:
    # spec carries: agent factory, prompt, deps, turn_timeout,
    # max_iters, telemetry recorder. Everything else is shared.
    ...

The face coordinator and notes coordinator become thin fan-out planners (which statements / which note sheets, how many Sheet-12 sub-agents) that hand specs to one runner. The ~400 lines of duplicated mechanics and the ~2,000 lines of parallel coordinator logic collapse to a single implementation. The monolith path is deleted, not parameterised.

4.3 — Typed tool contracts, no JSON strings

# Before: the model guesses a schema, emits a string, code re-parses
def fill_workbook(ctx, fields_json: str) -> str: ...

# After: pydantic-ai validates and injects the schema into the prompt
class FactWrite(BaseModel):
    concept: str          # resolved against the template's concept set
    period: Period        # CY | PY
    scope: EntityScope    # Company | Group
    value: float | None   # scalar facts
    html: str | None      # prose-note facts (mutually exclusive w/ value)
    evidence: Evidence    # {page:int, quote:str} — REQUIRED, not free text

async def write_facts(ctx, facts: list[FactWrite]) -> WriteReport: ...

Provenance becomes a required, typed field — directly enforcing invariant #2 instead of leaving "evidence" as an optional string the model may omit. Malformed proposals are rejected by the framework before the tool body runs.

4.4 — One fact-write interface, scalar and notes backends

Notes extraction is not a different kind of work — it produces facts about template rows, the same as scalar extraction — so it should go through the same write interface and the same runner. It is not the same physical table, and this distinction is load-bearing: scalar facts persist to run_concept_facts (whose value is REAL), while HTML-prose facts persist to notes_cells with their own cap and sanitisation invariants (gotcha #16). The live code already chose this exact shape — apply_fact branches on html and routes to _post_notes_fact (facts_api.py:253, the Phase 7 design pinned by tests/test_phase7_notes_unified.py). The rewrite's contribution is to extend that one interface up through the runner so the parallel notes/coordinator.py, the side-log JSON error channel, and the separate retry budgets disappear — not to collapse the two backends into one table. The notes-specific pieces (sanitisation, 30k rendered-char cap, cross-sheet dedup) survive as backend behaviour behind that interface.

4.5 — Orchestration as an explicit phase pipeline

Replace the 2,006-line generator with a declared sequence of phases, each a small unit with one job, one structured result, and one consistent error contract:

PHASES = [Validate, Extract, Cascade, Check, Review, Render]

async for phase in PHASES:
    result = await phase.run(ctx, sink)
    ctx.apply(result)
    if result.fatal:
        ctx.mark_terminal(phase.name); break
    sink.emit(StageComplete(phase.name))     # ONE event schema

This makes the lifecycle (gotcha #10's terminal-status guarantee) a property of the loop, not of try/finally discipline replicated per phase. SSE events get a single typed envelope. Reordering phases is editing a list.

4.6 — One error taxonomy

Three buckets, applied everywhere, replacing the 40+ ad-hoc swallows:

  • Advisory (telemetry, cost, trace save) — log, continue, emit a low-severity event. Allowed to swallow.
  • Recoverable (one statement fails, a cross-check throws) — degrade that unit, emit a typed ErrorEvent, complete the run as completed_with_errors.
  • Fatal (cannot write the fact store, cannot create the model) — abort, mark failed, re-raise to the phase loop.

The rule that distinguishes them is explicit, not implicit in which except happens to wrap the call.

5.1 — Kill "render twice" by precomputing routing

The v11 cross-sheet alias machinery (a single concept rendering at both a sub-sheet total and a face-sheet rollup coord) forces the importer, cell-resolver, exporter, reviewer, and concepts endpoint to each carry fallback logic — an 8-way test matrix per fact. The first-principles fix turns on a distinction the current model blurs: a coordinate is either a routing target (the exporter writes the fact's value there) or a read/review alias (it exists for agent-write resolution and Review-UI display, but the cell holds a live cross-sheet formula and must never receive an exported value).

Load-bearing invariant — do not collapse these

The face rollup cell (e.g. ='SOFP-Sub-CuNonCu'!B39) must stay a live formula; writing a literal there leaves Excel showing a stale snapshot and breaks the formula chain. This is pinned by test_exporter_preserves_cross_sheet_formula_on_alias_coord (tests/test_canonical_cross_sheet_rollup.py:305) and is the v11 contract in CLAUDE.md gotcha #21. An alias is therefore not a target, and folding aliases into concept_targets as if they were would regress exactly this test.

So the fix is: have the importer precompute the routing targets for every (period, scope) into concept_targets up front, so the exporter's value-routing is a single keyed lookup with no fallback. Aliases stay a distinct, clearly-named structure (the v11 concept_render_aliases table is fine) consulted only by cell-resolution and the Review UI — never by the value exporter. The win is that the exporter stops carrying three-way fallback for routing; it is not a license to make formula cells writable.

06What to keep, untouched

Do not throw the baby out

A first-principles rewrite is not a license to rebuild working subsystems. These are correct, well-tested, and encode hard-won domain knowledge. The rewrite reorganises around them.

  • The cascade engine (concept_model/cascade.py) — topological fixed-point recompute that respects aggregate_only and detects partial-state conflicts. This is invariant #3, correctly implemented.
  • The reviewer's no-plug guard (reviewer_agent.py:310) — deterministic, testable refusal of ungrounded writes and catch-all/abstract plugs (gotcha #17). This is exactly the "code disposes" principle done right.
  • Versioning / snapshot / revert (concept_model/versioning.py) — safety by reversibility rather than write-gating. Keep verbatim.
  • The cross-check framework (cross_checks/) — declarative, standard-aware checks. The consistency oracle the whole system needs.
  • The SSM template / linkbase derivation (gotcha #3) — never hand-edit; regenerate from the linkbase. Untouched.
  • Scout's calibration + TOC offset logic — genuinely hard, genuinely correct. Only the source-honesty flags get added.
  • truststore / UTF-8 / temperature constraints (gotchas #1, #5, and the Gemini-3 temp=1.0 rule) — environmental load-bearing facts, not slop.

Sequencing note

Steps 0–1 are mostly subtraction and should land first — they remove the code that would otherwise have to be migrated in every subsequent step. "Subtraction," not "free deletion": Step 0 retires a tested fallback path (so .env, the gotcha-#21 docs, and the fallback tests change with it), and Step 1's canonical_agent.py deletion is gated on first relocating the _load_open_conflicts helper the reviewer still imports. Audit every "delete" against current runtime imports before treating it as zero-behaviour. Step 4 is the keystone and the riskiest; it is safe to attempt only after the runner (2) and notes fold-in (3) have stabilised, because by then the fact store is exercised on every path and any drift between it and the xlsx is already caught by tests. Every step is independently revertable.

The one-sentence version

The system already contains the right idea — a typed fact store with a cascade engine and a grounded reviewer — but it was added beside three older designs instead of replacing them; the rewrite is the disciplined act of finishing that migration: commit to the fact store as the only truth, run every agent through one loop, and delete everything that competes.

Honest caveats on this report

Two places where I'd push back on my own findings before anyone acts on them:

  • Not all the "slop" is slop. Much of the defensive try/except exists because real Windows incidents (the EOFError race, gotcha #22; the truststore SSL fix, gotcha #5) taught the codebase to distrust its environment. The fix is to classify these handlers, not to remove them wholesale. Deleting a swallow without knowing which incident it commemorates is how regressions return.
  • Step 4 (store-first) is a genuine architecture change, not a refactor. It changes when and how the agent sees its own output, which can change extraction quality, not just structure. It needs an A/B against the current pipeline on the real test PDFs before it ships — "fewer lines" is not evidence of "same or better facts."
