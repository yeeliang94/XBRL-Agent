2 — The common prompt pattern

The iterative agents (the 5 face agents, the notes agents, the main scout loop) follow the same shape, which is what makes the caching story uniform:

Agent(model,
      deps_type=...,
      system_prompt=<one big pre-built string>,   # all static + dynamic content lives HERE
      model_settings=ModelSettings(temperature=1.0),
      history_processors=[strip_stale_images, ...])
  • The system prompt is the whole prompt. Static rules and run-dynamic data are concatenated into one system_prompt string at construction time. The first user message is usually a tiny near-constant trigger (e.g. "Extract the X… follow the strategy in your system prompt").
  • ModelSettings carries only temperature=1.0 — pinned by the "Temperature Constraint" rule in CLAUDE.md (CLAUDE.md:140; not gotcha #5, which is SSL). It exposes extra_headers and extra_body — the natural caching hook — never used.
  • For the iterative agents, PDF images are not in the prompt up front. They enter as view_pdf_pages tool results, then get stripped from later turns by strip_stale_images.

Exceptions to the pattern (don't over-generalize): the vision inventory fallback (scout/notes_discoverer_vision.py:339) is a one-shot agent that puts BinaryContent page images directly in the initial user prompt and uses a structured output_type, not a save-tool; the reviewer (correction/reviewer_agent.py:873) and the notes validator have no history_processors at all. So the "uniform shape" statement scopes to the iterative face/notes/scout-main agents only.

Configured today: nothing explicit

  • Repo-wide grep for cache_control / CachePoint / extra_headers / extra_body / anthropic-beta / cache_read_tokens / prompt_cache_key / prompt_cache_retention → zero hits in production code. 5 — The cost model from first principles

For one agent running T turns on a system prompt of size S, input cost is approximately:

input_tokens  ≈  T·S  +  Σ(growing history)

The T·S term dominates when S is large and fixed — exactly this codebase (multi-thousand-token prompts; T up to 40 for face/notes, up to 36 for the reviewer). This yields a strict lever order:

  1. Reduce T — kills input and output cost and latency at once. Highest theoretical lever.
  2. Reduce S — linear win on the dominant term.
  3. Cache the residual S — turns T·S into S + (T−1)·0.1·S.

Caching is the cheapest and least risky action, so it's the right first move — but it is the smallest lever. It makes waste cheap rather than removing it. Don't cache a bloated prompt and declare victory.

The key insight: strip and cache are complementary, not competing. Assign by content type:

  • Needed re-sent every turn (rules, persona, tool schemas, template structure) → cache it. Can't drop it.
  • Not needed re-sent (stale images, duplicate template) → strip it. ~100% reduction on all providers, beats caching's ~90%.

So strip_duplicate_template is correct and superior to caching for that content. The only constraint is ordering: a strip that mutates a message above the cache breakpoint invalidates the cache. Since strips touch tool-return messages (after the system prompt) and the natural breakpoint is end-of-system-prompt, they're compatible by construction.

6 — Recommendations

  1. Measure before touching anything. P0

    Pull pydantic-ai's normalized cache_read_tokens and cache_write_tokens from usage() into the live path and onto the Telemetry tab — extend _in_tokens (server.py:110), which today reads only input_tokens. Capture writes too: Anthropic bills cache writes at a premium, so a telemetry that counts only reads will report phantom savings and misprice write-heavy runs. Costs nothing at runtime and answers the question nobody can currently answer: is OpenAI auto-caching even hitting today? On the default path it may already be saving most of T·S — or 0% if the prefix is unstable. You are currently making a cost decision blind.

  2. Anthropic caching — path-dependent, not a one-liner. P0

    Every Claude run (incl. the reviewer on a Claude model) pays full T·S with no caching, but the fix splits by route:

    • Direct mode (native AnthropicModel): set the cache breakpoint via pydantic-ai's Anthropic cache settings / CachePoint after the static system prefix. Confirm the exact 1.77 API against the installed build first — local env is stale 0.8.1.
    • Proxy mode (the default on Mac; Claude routes as OpenAIChatModel through LiteLLM): the breakpoint must be expressed as OpenAI-format cache_control on content blocks and survive LiteLLM's translation to Anthropic. Verify drop_params: true (litellm_config.yaml) doesn't strip it; this likely needs proxy-config work, not just agent settings.

    So "one change at _create_proxy_model" was wrong — it's two mechanisms gated on proxy_url.

  3. Use OpenAI's explicit cache controls on the default path. P1

    OpenAI auto-caches prefixes, but two documented controls sharpen it and the repo uses neither. prompt_cache_key (a routing hint) keeps requests with the same prefix on the same cache shard — set it per agent-type so the 5 face agents / N Sheet-12 sub-agents land on consistent caches. prompt_cache_retention (extended retention, ~24h) directly addresses the cross-run TTL limit that otherwise caps reuse to a few minutes. Both pass via extra_body on the OpenAI path. Cheap, and they compound with the telemetry from rec 1.

  4. Look harder at PDF images than at text caching — likely the bigger fish. P1

    The same PDF pages are rendered to images and sent to the model by the scout, then ~5 face agents, ~5 notes agents, then the reviewer. tools/page_cache.py caches the PNG rendering (CPU) but not the vision tokens — each agent re-pays the model for the same image. Within an agent, strip_stale_images handles repeats; across agents, nothing does. Cross-agent image caching is hard (independent prefixes), so the real lever is upstream: a richer scout extraction (face_line_refs, inventory, context) means fewer downstream view_pdf_pages calls — cutting vision tokens and T. Measure the vision-token share per agent before micro-optimizing text caching.

  5. Re-evaluate the dead cache_template path — opportunity, not just cleanup. P1

    Embedding the template summary in the system prompt (vs a read_template tool call) saves a full turn per agent and makes the block cacheable. The catch: it inflates turn-1 cost. Pre-caching, tool-result + strip was right; post-caching, the calculus may flip because turn-1 cost amortizes while the saved turn is pure win. Re-run the numbers once telemetry exists. Either wire it properly or delete the dead parameter — don't leave a trap someone sets expecting it works.

  6. Move the Sheet-12 inventory slice to the prompt tail — but warm the cache first. P2

    The N fan-out sub-agents share an identical large prompt except === INVENTORY ===, rendered mid-prompt. Moving it to the end (the assigned-note list already lives in the user message) makes the full static prefix eligible to cache across the burst. Caveat (peer review): the sub-agents launch concurrently with only a 0.6s stagger (_SUB_AGENT_LAUNCH_STAGGER_SECS, notes/listofnotes_subcoordinator.py:82), and a cache entry isn't readable until the first response that wrote it begins — so "first writes, N−1 read" assumes a warm cache that concurrency defeats. To realize the saving you'd need a deliberate warm-up (run one sub-agent to first-token before fanning out) or serialization, proven by the rec-1 telemetry. Scope honestly: this is a cross-agent (scope-B) optimization only; the dominant within-agent T·S caching warms naturally and is unaffected. The same warm-up caveat applies to the 5 parallel face agents and to the "second breakpoint after _base.md" idea in the next rec.

  7. Make the static/dynamic boundary a code boundary, not a string concat. P2

    Today static rules and dynamic data are jammed into one system_prompt string (the reviewer's body + packet is the clearest case). For within-run caching this is fine — built once, stable across turns. The reason to separate is robustness: with static content as system_prompt and dynamic content as the first user message, the cache breakpoint falls naturally at the system/user boundary and can't be accidentally broken by editing the packet. Unlocks a second breakpoint after shared _base.md so it caches across the 5 parallel face agents (small, but free once you're placing breakpoints).

  8. Audit prompt assembly for non-determinism. P2

    Cache keys hash the byte-exact prefix. Unsorted dict/set iteration, inconsistent float formatting, or run-varying data in the static region produces a different prefix and a guaranteed miss. Invisible within a run (built once); across runs and the 5-agent fan-out it's hit-vs-miss. Worth a deliberate pass once telemetry shows the real hit rate.

  9. Know the input/output split per agent — don't apply uniform effort. P2

    Caching only touches input. Face extraction is input-heavy (big prompt, small writes) → high cache ROI. Notes agents emit up to 30k-char HTML cells → output-heavy → caching does little. The telemetry from step 1 should split this so effort goes where input dominates.

Sequencing

Telemetry (1) → Anthropic breakpoint (2) → read the numbers → let the data decide whether the real money is in vision tokens (3), turn reduction (4), or fan-out caching (5). Steps 1–2 are small, contained edits at known sites. Everything after 2 should be telemetry-driven — including, honestly, whether further caching work beats simply reducing T.

9 — Effectiveness problems (accuracy, not cost)

A separate axis from caching: defects in the content of the prompts that cause the agents to extract wrong numbers, mis-route disclosures, flail, or run out of budget. Severity reflects likely impact on output correctness. Tags: HIGH MED LOW.

Why this matters more than it looks. The notes pipeline is deliberately all-LLM-judgement — no deterministic matching, no synonym dictionary, no downstream corrector (gotcha #14) — and extraction has only a weak verifier. In this architecture the prompt is the program: a defect converts directly into a wrong number or a dropped disclosure with nothing to catch it. Prompt quality is the largest single effectiveness lever in the system, ahead of model choice.

What already works (not flagged). strip_stale_images is stage-aware — keeps all images until the first write, strips only pre-write batches; it already fixed the run-126 re-fetch loop (extraction/history_processors.py:158). The reviewer packet's comparands + entity_scope tags are good. The scout scale_unit 1000×-error mitigation is exemplary. The vision notes prompt is tight. Evidence-required-per-fact grounding is sound.

2 — Hard-coded coordinates the agent is told to trust but can't verify HIGH

Violates the codebase's own label-based, regeneration-survivable design (gotcha #3/#15). When a template regenerates, the prompt lies.

  • SOCIE asserts rows 6–25 (CY) / 30–49 (PY) and "this is a matrix template — not label matching" (socie.md:40) — i.e. trust the literals, don't verify. But the group overlay uses 3–25/27–49 and gotcha #12 confirms group blocks start at row 3. Off-by-one in the live template → every SOCIE write lands wrong, with no label-match safety net. Highest-risk single instruction in the face set, and the 24-column matrix ships with one bare example.
  • Notes templates hard-code "row 57", "rows 6–33", "27 data-entry rows" (notes_issued_capital.md, notes_related_party.md).

Fix: read movement-row numbers from read_template() rather than trusting literals; add one worked example per movement type (profit, dividend, share issue, OCI); reference labels, not absolute rows, in the notes prompts.

3 — The feedback loop gives the agent no gradient to act on HIGH

An agent only improves if each turn yields actionable signal. Today a careful agent flails:

  • verify_totals is near-vacuous for 4 of 5 statements — for SOPL/SOCI/SOCF it checks formula-driven identities that tie whenever numbers are consistent; it never compares to the PDF (verifier.py:1169; sopl.md says "balance checks are SOFP-only"). The save-gate treats passing verify as license to save, so an agent filling consistently-wrong numbers sails through believing it is "verified." The prompts never warn that verify ≠ correctness here.
  • When verify fails, feedback is non-directive by design (gotcha #17, to prevent plugging) — but for SOCF the commonest error is a sign flip, and the feedback holds computed_totals yet never says "this gap matches row X — likely a sign error." The no-plug agent gets a failure with no direction and is told not to do the obvious thing → burns turns re-verifying with no new information.
  • Some prompts tell the agent to cross-check against statements it doesn't have (soci.md:38, socf.md:48) — those checks run later on merged workbooks; the face agent has no SOFP to compare and may stall.

Fix: give directional diagnosis without prescribing a plug ("equity side light by exactly the PPE figure — recheck that row's sign"); tell the agent verify is vacuous for non-SOFP and value-accuracy is its own responsibility; drop the impossible cross-checks from face prompts.

4 — Context starvation against the turn budget: the careful path is the expensive path HIGH (reviewer)

The reviewer cap is 12 + 2·n_items, clamped [12, 36] (reviewer_agent.py:823), and turn_count increments per model→tools round-trip. But the prompt's own playbook prescribes ~6 round-trips per failing check (list_facts → trace lhs → trace rhs → read leaf → view PDF → apply_fix). With n_items=4 the cap funds ~3 checks while the careful path needs ~6 each. The agent that obeys the prompt's discipline exhausts first — on exactly the hard runs the reviewer exists for. Two compounding causes:

  • The cascade trace is computed server-side already (reviewer_agent.py:224) but withheld from the packet, forcing 2–3 rediscovery rounds per check.
  • The prompt reads as a serial recipe and never says "batch independent tool calls in one turn" — pydantic-ai supports parallel calls.

Fix (highest-ROI reviewer change): pre-inject the trace (children + signed coefficients + children-sum-vs-parent delta) into _format_review_packet — zero model cost, directly buys back the budget; and instruct tool-call batching.

5 — Confident wrong hints are worse than empty ones HIGH (scout)

In a soft-hint design the value of a hint is P(correct)·benefit − P(wrong)·misdirection. The scout prompt inverts the sign on the hardest inputs: on scanned PDFs it is told to "populate face_line_refs yourself" from the image — transcribe every line-item→note-number mapping — with no confidence gate and no "leave note_num null if illegible," despite applying exactly that "do NOT guess" discipline to face-page location (scout/agent.py:155,175). A confidently-wrong note number sends the extraction agent to the wrong note pages first. Compounding: save_infopack reports "saved successfully" even when it silently dropped malformed inventory/ref entries (scout/agent.py:624,700), so scout "succeeds" with an empty inventory and never self-corrects.

Fix: gate scanned-PDF refs on high confidence and null illegible note_nums; return surviving counts from save_infopack ("3 statements, 14 notes, 22 refs; 2 skipped — re-check") so the agent retries in-run.

6 — Positioning + repetition crowd out the task MED (pervasive)

LLM attention favours start and end; this codebase fills the middle with generic sermons and repeats them. The no-plug rule appears 4× (base, statement file, verifier footer, save-gate message) while the statement-specific "fill the sub-sheet first, here are the exact rows" sits buried mid-prompt. The Sheet-12 sub-agent's assigned batch is in a mid-prompt block labelled generically "Scout identified N notes" (notes/agent.py:561) — not "YOUR ASSIGNED BATCH" — and also in the user message in a different format, so the agent reconciles two representations of its own scope (double-write / missed-note risk).

Fix: state shared rules once; move per-agent decision-critical content (assigned batch, target layout) to the end; relabel the inventory block as the batch.

7 — Temperature 1.0 forced globally on a precision task MED — verify per model

Every agent hardcodes ModelSettings(temperature=1.0). The reason is a real Gemini-3-through-proxy constraint — the unnumbered "Temperature Constraint" rule in CLAUDE.md (CLAUDE.md:140; not gotcha #5, which is SSL) — but it is applied to all providers, including the default openai.gpt-5.4 and any Claude reviewer. Financial extraction is a low-entropy transcription/arithmetic task; 1.0 maximises sampling variance.

Fix: make temperature provider-aware — force 1.0 only for Gemini, drop to ~0–0.2 elsewhere. Caveat: some GPT-5 reasoning models only accept the default temperature, so this may be a no-op on OpenAI but a real win on Anthropic — validate per model rather than assume.

8 — Absolute rounding tolerance manufactures unresolvable imbalances MED

The SOFP balance check uses abs(diff) > 0.01 in absolute RM (verifier.py:482). On an RM'000 statement, a legitimate ±RM1 source-rounding difference becomes a hard imbalance the agent is told not to plug → straight into the burn-turns-then-acknowledge loop. Fix: scale the tolerance to the statement's unit/magnitude.
