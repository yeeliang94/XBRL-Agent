House rule reminder

Per CLAUDE.md, every item that brushes a load-bearing invariant must land with its pinning test in the same commit, and "done" means the named test passes — not "looks right". Each card below names its tests for exactly that reason.

AScout hardening

The scout feeds page hints, face-line refs, the notes inventory, and entity context to every downstream agent — yet it is the only agent with no per-turn timeout, no wall-clock cap, no trace persistence, and several silent-coercion paths. These five items bring it to parity with the face/notes harnesses.

1 Scout per-turn + wall-clock timeout Done 2026-06-12Effort MPhase 2

Problem / evidence

run_scout_streaming (scout/agent.py ~L1188) iterates agent.iter() with only a between-turns check against MAX_AGENT_ITERATIONS. A single stalled model request hangs the scout — and the run — indefinitely. This is the exact failure mode NOTES_TURN_TIMEOUT (notes/coordinator.py:128) was added to fix for notes agents. The SSE endpoint (api/uploads.py:128–246) has no server-side timeout either.

Concrete change
  1. Migrate the scout's iteration loop onto agent_runner.iter_with_turn_timeout (or fully onto run_agent_loop with a scout AgentLoopSpec): add SCOUT_TURN_TIMEOUT = 180.0 module constant in scout/agent.py.
  2. Wrap the whole scout run in a monotonic wall-clock check mirroring server._run_reviewer_pass: XBRL_SCOUT_WALLCLOCK_S env override, default 300, 0 disables (same resolver semantics as XBRL_CORRECTION_WALLCLOCK_S).
  3. On expiry, emit a structured SSE error event ("scout timed out — run can proceed without hints") and return a degraded-but-valid empty Infopack, since the whole pipeline already degrades gracefully on missing scout output (gotcha #13).
  4. Apply the same wrapper to the non-streaming run_scout path (scout/agent.py ~L1131–1185) used by the CLI.
Reuse

agent_runner.iter_with_turn_timeout / AgentLoopSpec (rewrite Phase 2); the wall-clock resolver pattern in server.py:903–918.

Pinning tests

New tests/test_scout_turn_timeout.py mirroring tests/test_notes_turn_timeout (stalled-model stub → structured timeout, not hang); a wall-clock case with a tiny cap. Existing scout suites (test_scout_face_line_refs_wiring.py, test_scout_populates_context.py) stay green.

Acceptance criteria

A scout whose model never responds terminates within SCOUT_TURN_TIMEOUT + ε with a structured SSE error, and the upload page shows the failure instead of an eternal spinner.

Invariant notes

Gotcha #18: turn caps must stay below pydantic-ai's silent 50 — unchanged here; the scout already uses MAX_AGENT_ITERATIONS. Gotcha #13: timeout fallback must NOT introduce page restrictions — it returns an empty infopack, never a partial "allowed pages" set.

2 Persist scout conversation trace Done 2026-06-12Effort SPhase 2

Problem / evidence

The scout never calls save_agent_trace — when it misses a statement, mislabels a variant, or hallucinates a page there is no transcript to debug. Face and notes agents persist traces on success and failure (gotcha #6: "the trace viewer is useful exactly when debugging a failure"); the scout is the one agent that doesn't.

Concrete change
  1. Wire the audit context into the scout entry point first. Today run_scout / run_scout_streaming (scout/agent.py:1131 / :1188) take no output_dir / run_id — they only return an Infopack. Add an output_dir kwarg (and run_id if the SCOUT row of step 3 is taken) threaded from the coordinator's scout launch site, so the trace has a destination. Without this the next step has nowhere to write.
  2. On scout completion in run_scout_streaming, call agent_tracing.save_agent_trace(result, output_dir, "SCOUT", turns=…) writing scout_conversation_trace.json into the session output dir.
  3. On the new timeout / exception paths (item 1), use the best-effort save_messages_trace fallback from agent_run.ctx.state.message_history — same pattern the face coordinator uses for partial runs.
  4. Make the trace reachable — the path check is not enough. GET /api/runs/{id}/agents/{stmt}/trace rejects any statement not in the run's run_agents rows BEFORE the path-containment check (api/runs.py:260–265: known = {a.statement_type …} → 404). So either (a) create + finalize a SCOUT run_agents row at scout launch (preferred — also lets History/Telemetry list the scout), or (b) special-case "SCOUT" into the known whitelist. The path-containment check generalises once SCOUT clears that gate, not before.
Reuse

agent_tracing.save_agent_trace / save_messages_trace (100 KB payload cap, binary elision) — no new serialisation code.

Pinning tests

Extend tests/test_agent_tracing.py: scout success writes the file; scout failure writes the partial-history fallback. Hybrid-storage rule asserted (trace on disk, never in SQLite). Plus a route-reachability test (tests/test_run_routes.py or sibling): after a scout run, GET /api/runs/{id}/agents/SCOUT/trace returns 200 — proving the run_agents whitelist gate of step 4 was actually opened, not just that the file exists on disk.

Acceptance criteria

Every scout invocation — success, timeout, crash — leaves scout_conversation_trace.json in the session dir.

Invariant notes

Gotcha #6 hybrid storage: verbatim content stays on disk, not the DB. Honour the 100 KB per-payload cap (scout turns include page images — the binary-elision path matters).

3 Loud warnings on infopack coercions Done 2026-06-12Effort SPhase 2

Problem / evidence

_save_infopack_impl (scout/agent.py ~L785–787) and Infopack.from_json (scout/infopack.py ~L337–358) silently coerce invalid scale_unit / consolidation_level / detected_standard values to "unknown". Given that a wrong scale unit "silently inflates every extracted value" (gotcha #13's own wording, sibling of #17), a coercion is a signal worth surfacing, not swallowing.

Concrete change
  1. At each coercion site, logger.warning("scout emitted invalid %s=%r — coerced to 'unknown'", field, raw).
  2. Add a coerced_fields list to the survival-count telemetry block the save already emits (scout/agent.py ~L827–844) so degradation is visible in the SSE stream / logs, not only server-side.
  3. Same treatment in from_json's malformed-entry drops (~L233–335): count dropped face refs / inventory entries into the same telemetry summary.
Reuse

The existing survival-count emission — extend its payload rather than inventing a new event type (the frontend SSE filter only recognises the fixed vocabulary; see the warnings-on-complete precedent in notes/coordinator.py).

Pinning tests

New cases in the infopack suite (tests/test_infopack_context_schema.py area): invalid scale_unit → coerced AND warning logged (use caplog) AND counted in the telemetry dict.

Acceptance criteria

No silent coercions remain: every coercion or dropped entry is visible in logs and in the save-time telemetry summary.

Invariant notes

Coercion behaviour itself is unchanged (defensive deserialisation stays — gotcha #13's graceful degradation); only the observability changes.

4 note_num sanity bounds Done 2026-06-12Effort SPhase 2

Problem / evidence

FaceLineRef.note_num is validated only as ≥ 1 (scout/infopack.py:62–71); the vision inventory schema allows 1–999 (scout/notes_discoverer_vision.py:58 Field(ge=1, le=999)). A hallucinated "Note 743" on a 30-note filing sends a face agent hunting for a note that doesn't exist.

Concrete change
  1. Add a module constant MAX_PLAUSIBLE_NOTE_NUM = 150 in scout/infopack.py.
  2. In _save_infopack_impl, drop (with a warning, feeding item 3's telemetry) any LLM-emitted FaceLineRef whose note_num exceeds the bound — or, when the deterministic inventory is non-empty, exceeds max(inventory note_num) + 5 (tighter, evidence-based bound).
  3. Tighten the vision schema to le=150; keep the Field(ge=1, …) contract gotcha #13 documents so the int(item["note_num"]) coercions in notes/coverage.py:256 are untouched.
Reuse

Existing drop-with-warning machinery in from_json; item 3's telemetry list.

Pinning tests

Extend tests/test_scout_face_line_refs_schema.py: ref with note_num=743 dropped + warned; ref at the boundary kept. Vision schema test for the new ceiling. tests/test_sheet12_ignores_subnotes.py stays green (sub-note refs are strings, unaffected).

Acceptance criteria

No infopack can carry a face-line ref pointing at a note number beyond the plausible bound; drops are visible in telemetry.

Invariant notes

Gotcha #13: hints stay advisory — this filters obviously-invalid hints, it does not restrict pages or enforce hint usage. The negative assertions in tests/test_page_hints.py must stay green.

5 Explicit scanned-PDF signal from face-structure reads Done 2026-06-12Effort SPhase 2

Problem / evidence

On scanned PDFs the deterministic regex (scout/face_structure.py:85–152) returns [] silently. The scout's prompt tells it to populate face_line_refs itself in that case, but nothing tells the LLM the regex came back empty, and nothing detects when the LLM also fails to populate — downstream face agents just silently lose the structural hint.

Concrete change
  1. _read_face_structure_impl (scout/agent.py ~L309–354): when the regex yields zero refs, include "scanned_hint": true and one explicit sentence in the tool's JSON return ("no text layer found — likely a scanned page; populate face_line_refs from the page image yourself").
  2. In _save_infopack_impl: when the regex was empty AND the LLM submitted no refs for a statement that was read in detail, log a warning and count it in item 3's telemetry ("face refs unavailable for SOFP — scanned page, LLM did not populate").
Reuse

The existing face_read_in_detail flag and the cached-regex-vs-LLM merge logic at scout/agent.py ~L620–681.

Pinning tests

Extend tests/test_scout_face_line_refs_wiring.py: empty-regex path returns the hint field; save-time warning fires when both sources are empty. Prompt-render tests (test_prompts_render_scout_face_refs.py) unchanged — empty refs still fall through to the bare hint block.

Acceptance criteria

A scanned-PDF scout run either produces LLM-populated face refs or a visible warning naming the statements that lost them — never a silent gap.

Invariant notes

Gotcha #13's graceful degradation is preserved: empty refs remain valid output. The confidence-gating from PLAN Phase 8 (scanned face-refs survival counts) is the adjacent code — extend, don't fork, its counters.

BExtraction hardening

The face loop already has a 180s per-turn timeout, a 40-iteration cap, and timeout/iteration-cap salvage paths. What it lacks: an overall time ceiling, a spend ceiling, transient-error tolerance, atomic file writes, and a failure taxonomy.

6 Face-agent wall-clock cap Done 2026-06-12Effort S–MPhase 3

Problem / evidence

coordinator.py::_run_single_agent (~L473–615) bounds each turn (180s) and the turn count (40) but not the total: 40 slow-but-compliant turns is legally ~2 hours per agent. The correction and notes-validator passes both received wall-clock caps (gotcha #18); the face agents — the longest-running components — never did.

Concrete change
  1. Add FACE_WALLCLOCK_TIMEOUT (default 1800.0) to coordinator.py, resolved via XBRL_FACE_WALLCLOCK_S with the same positive/zero-disables semantics as XBRL_CORRECTION_WALLCLOCK_S (server.py:903–918).
  2. Check time.monotonic() against the cap at the top of each loop iteration in _run_single_agent (the same in-loop placement as server._run_reviewer_pass ~L1136).
  3. On expiry, route through the existing salvage logic (coordinator.py ~L776–858): if deps.filled_path exists and the last verification was clean → status succeeded with reason wallclock_after_write; else failed with a structured message. Save the partial trace via the existing best-effort helper.
Reuse

The timeout/iteration-cap salvage branches (do not write a third copy — extract a small _salvage_or_fail(reason) helper used by all three exits); the wall-clock env resolver from server.py.

Pinning tests

New tests/test_face_wallclock_cap.py: tiny cap + slow-stub model → terminal result within bound; salvage case (workbook written + clean verify) lands succeeded; pre-write case lands failed. tests/test_max_agent_iterations_below_pydantic_cap.py unaffected.

Acceptance criteria

No face agent can run past the cap; expiry after a clean write keeps the user's workbook (mirrors the Stop-All partial-merge philosophy, gotcha #10).

Invariant notes

Gotcha #18: this is additive defence-in-depth — the 40-turn cap and 180s turn timeout stay exactly as-is.

7 Per-agent token budget Done 2026-06-12Effort SPhase 3

Problem / evidence

Iterations are bounded; tokens are not. A verbose model on a large PDF can burn millions of tokens within 40 legal turns — runaway cost with no early exit. The per-turn cumulative-usage delta is already computed for v8 telemetry (gotcha #6), so the measurement exists; only the ceiling is missing.

Concrete change
  1. In agent_runner.run_agent_loop, accumulate the per-turn total-token delta it already derives; compare against a new optional AgentLoopSpec.token_budget.
  2. Resolve the budget from XBRL_MAX_TOKENS_PER_AGENT (default 0 = disabled — opt-in first, flip the default after observing real run costs).
  3. On breach, raise a structured TokenBudgetExceeded(RuntimeError); map it in _run_single_agent to the same salvage-or-fail handling as IterationLimitReached, with reason token_budget_exceeded.
Reuse

The v8 per-turn usage delta in run_agent_loop; the IterationLimitReached exception/handling pattern; item 6's _salvage_or_fail helper.

Pinning tests

New cases in the agent-runner suite: budget breach → structured exception at the next turn boundary; budget=0 → never fires; salvage mapping in the coordinator test alongside item 6's.

Acceptance criteria

With a budget set, an agent stops within one turn of crossing it and reports token_budget_exceeded in its result (and item 9's error_type).

Invariant notes

Gotcha #6: per-turn token figures are best-effort deltas — the budget check tolerates that (it bounds spend approximately, which is its job; exactness is not required).

Gotcha #22 fixed the notes validator with _atomic_save_workbook (tempfile + os.replace) after the 2026-05-29 Windows EOFError incident, and explicitly flags the same load→save-in-place pattern as latent in tools/fill_workbook.py and concept_model/exporter.py.   3. Audit remaining live-path savers (notes/writer.py, workbook_merger.py) and convert them in the same pass — one mechanism, zero stragglers.
Reuse

The proven helper itself — atomic on Windows + POSIX, already battle-tested by tests/test_notes_validator_agent.py::TestWorkbookIoRaceSafety.

Pinning tests

TestWorkbookIoRaceSafety stays green against the moved helper. New unit test: fill_workbook writes via os.replace (monkeypatch-spy), and a mid-save-interrupt simulation leaves the old file intact.

No live code path saves a workbook with in-place wb.save(); a reader concurrent with any save sees old-or-new, never a truncated zip.

Invariant notes

Directly closes gotcha #22's "latent elsewhere — don't reintroduce the pattern" warning. Update that gotcha's text when this lands.

9 Structured error_type taxonomy (schema v17) Done 2026-06-12Effort MPhase 3

Problem / evidence

Every agent failure collapses into one free-text error string in run_agents. Post-mortems and the Telemetry tab can't distinguish "DB projection failed" from "save-gate never opened" from "turn timed out" without string-grepping.

Concrete change
  1. db/schema.py: bump CURRENT_SCHEMA_VERSION to the next available version (16→17 as of this writing — use whatever the committed value + 1 is when this lands, since another migration may merge first); add nullable run_agents.error_type TEXT via the matching _Vn_MIGRATION_COLUMNS tuple (additive ALTER, follows the v15 columns pattern). No CHECK constraint — same rationale as runs.status.
  2. Define the vocabulary as constants next to AgentResult: turn_timeout · iteration_capped · wallclock · token_budget_exceeded · projection_failed · save_gate_refused · tool_exception · cancelled · no_write.
  3. Add an error_type: str | None = None parameter to repo.finish_run_agent (db/repository.py:484) and its UPDATE — it has no such column today; defaulting to None keeps the many legacy call sites working (same compatibility pattern as the v8 telemetry rollups already on that signature).
  4. Set it at every failure branch that finishes a row, not just the face coordinator: coordinator.py::_run_single_agent, the notes coordinator's terminal results, AND the pseudo-agents that also call finish_run_agent — the CORRECTION/reviewer row (api/reviewer.py _finalize_correction_agent_row) and NOTES_VALIDATOR (and SCOUT once item 2 creates its row). Otherwise the acceptance criterion ("every failed/cancelled agent row") is unmet and those rows carry NULL.
  5. Surface read-only in the run-detail Agents tab + History API payloads (additive field; frontend renders a small badge).
Reuse

The v11–v16 migration-step pattern (idempotent walk-forward, nullable column); salvage-reason strings from items 6/7 double as error_type values.

Pinning tests

New tests/test_db_schema_v17.py (fresh-init + v16→v17 walk, per the series); coordinator test asserting each failure branch writes its taxonomy value.

Acceptance criteria

Every failed/cancelled agent row carries a non-null error_type; History can be grouped by failure class.

Invariant notes

Gotcha #11: column must be nullable (SQLite ALTER constraint); old DBs walk forward one version at a time; each step idempotent.

10 Transient-error retry for face agents Done 2026-06-12Effort MPhase 3

Problem / evidence

Notes agents retry once on any non-cancellation error plus a separate 3-deep rate-limit budget with honoured retry-after (notes/coordinator.py:510–646). Face agents have zero retries (coordinator.py:884–905): one transient proxy blip or 429 fails the whole statement.

Concrete change
  1. Wrap the single-attempt body of _run_single_agent in a retry loop that fires only for errors classified transient: notes/_rate_limit.is_rate_limit_error (429s, honouring compute_backoff_delay with up to RATE_LIMIT_MAX_RETRIES) and connection-class errors (httpx.ConnectError/ConnectTimeout) with exactly one retry.
  2. Generic exceptions keep today's fail-fast behaviour — face retries re-bill a large PDF context, so no blanket retry budget.
  3. Schedule backoff sleeps inside the try so a user abort during backoff lands on the CancelledError branch (the notes loop's pending_backoff pattern, verbatim).
  4. Emit the existing retry-marker status event style ("retrying (attempt 2) — last error: …") so operators see it live.
Reuse

notes/_rate_limit.py wholesale (classification, backoff, budget constants); the notes coordinator's dual-budget loop as the structural template.

Pinning tests

New tests/test_face_transient_retry.py (failure-injection, modelled on tests/test_notes_retry_budget.py): 429 → backoff → success; ConnectError → one retry; ValueError → no retry; cancel-during-backoff → status cancelled.

Acceptance criteria

A single transient provider failure no longer fails a statement; a real code error still fails fast on first occurrence.

Invariant notes

Keep the retry whole-attempt (fresh agent + deps) like the notes loop — never resume a half-run conversation. The DB projection (gotcha #21 Phase B) is idempotent per write-batch, so a retried attempt re-projecting is safe; verify with the existing projection tests.

CReviewer hardening

The reviewer loop itself (dynamic turn cap, 180s turns, 300s wall-clock, snapshot-before-write, durable run_review_tasks) is sound. These items close the consistency windows around its edges — where the DB, the workbook on disk, and the user's mental model can drift apart.

11 Surface cascade failures (revert + post-pass) Done 2026-06-12Effort SPhase 4

Problem / evidence

After "Revert to original", concept_model/versioning.py (~L178–180) runs recompute_after_turn outside the transaction and swallows any exception — the user sees "reverted ✓" while totals are stale and no conflict rows flag it. The reviewer's post-pass re-cascade (server.py ~L1266–1272) swallows identically.

Concrete change
  1. versioning.revert_to_original: catch the recompute exception, return cascade_ok: false + cascade_error: str in its result instead of suppressing.
  2. api/reviewer.py revert endpoint: propagate those fields; the response stays 200 (facts ARE restored) but carries the warning.
  3. server._run_reviewer_pass: record outcome["cascade_error"] on the post-pass recompute failure path.
  4. web/src/components/ReviewTab.tsx: render a warning banner when either field is set ("values restored, but totals could not be recomputed — re-run review or re-check").
Reuse

The Review tab's existing flag/banner styling (theme tokens from web/src/lib/theme.ts — gotcha #7, inline styles only).

Pinning tests

Extend tests/test_reviewer_versioning.py: recompute raises → cascade_ok=false returned, facts still restored. Route test in tests/test_reviewer_routes.py for the surfaced field. Frontend test for the banner.

Acceptance criteria

A cascade failure after revert or review is visible to the user within the same response — never a silent stale-totals window.

Invariant notes

Keep the recompute outside the revert transaction (deliberate — it opens its own connection); only the reporting changes. Gotcha #21's revert guarantee ("restore with one click") is strengthened, not altered.

12 Stale-download marker after failed re-export Done 2026-06-12Effort SPhase 4

Problem / evidence

After a manual re-review, _reexport_remerge_durable is best-effort: if it fails, run_concept_facts has moved but filled.xlsx hasn't — the Review tab's diff (DB-fed) is correct while the download is stale, with no signal. The inline pass has canonical_export_degraded for per-statement export failures; the re-review path has nothing.

Concrete change
  1. (Already done — do not re-do.) _reexport_remerge_durable already returns bool (server.py:4307); it does not swallow the result. The actual gap is the consumer: the manual re-review path calls it and discards the return (api/reviewer.py:270: if outcome.get("writes_performed",0)>0: server._reexport_remerge_durable(run_id)). Capture that bool and, on False, write export_stale: true + the error into the re-review outcome dict persisted to run_review_tasks.outcome_json.
  2. GET /api/runs/{id}/re-review/status already returns the outcome — the field rides along for free.
  3. Prior art to cite, not duplicate: the download endpoint already fail-closes when a re-export fails and post-run manual edits would otherwise be served stale (api/files.py:197+). This item adds the Review-tab signal for the re-review-write case; it is not a second fail-safe on the download.
  4. Review tab: badge "download may be stale — re-export failed" with a retry affordance (re-call the durable re-export; endpoint can be the existing download path's re-export-from- facts, which already regenerates on demand).
Reuse

run_review_tasks durability (schema v13) — no new table; the download endpoint's existing re-export-from-facts fallback (server.py ~L360–390) is the natural retry mechanism.

Pinning tests

Extend tests/test_reviewer_routes.py: forced re-export failure → export_stale in status payload; outcome survives restart (rides the existing simulated-restart test).

Acceptance criteria

DB↔xlsx divergence after a re-review is impossible to miss from the Review tab.

Invariant notes

Gotcha #21's "no xlsx split-brain" intent; gotcha #10's never-double-fault rule — the failure write itself stays inside the swallowing _save_review_task (terminal writes are best-effort by design).

13 Snapshot race fix Done 2026-06-12Effort SPhase 4

Problem / evidence

The reviewer's re-entrancy guard covers run_review_tasks, but the has_snapshot() check and snapshot_facts() write (server.py ~L1042–1059, concept_model/versioning.py:45–89) use two separate connections. Two concurrent passes can interleave between check and write, and a second snapshot would overwrite the first — destroying the original-extraction restore point.
