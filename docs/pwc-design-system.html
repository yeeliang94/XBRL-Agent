Application UI Standards

XBRL Agent Design System v3

The behavioural and visual specification for the web UI. It keeps the PwC black-and-orange identity while strengthening accessibility, semantic hierarchy, financial-data density, responsive behaviour, and component governance.

Overview

This document defines the visual language, task behaviour, responsive expectations, accessibility floor, and governance model for the application. It is deliberately optimized for evidence-led financial work rather than a generic SaaS dashboard.

Governance: decisions begin here, flow through semantic tokens and component primitives, and end in tested product behaviour. Accessibility and responsive behaviour are part of every component contract, not a final polish pass. The CSS variables here exist only to render this document.

Rules for AI developers

The load-bearing conventions. Adhering to the design system means following these — not just picking nice colors.

01Inline values; narrow state hooks. Layout, colour, type, spacing, and geometry use inline style={{…}} objects. Do not use Tailwind or a class-based utility system. A shared uiClass.* hook is allowed only for hover, focus, animation, editor styling, or another state inline styles cannot express.
02Choose meaning, not a hue. Components consume semantic roles such as primary action, muted text, subtle border, and success icon. Raw palette tokens are implementation ingredients, not component decisions.
03Reuse the primitives. Spread ui.buttonPrimary, ui.card, ui.input… from lib/uiStyles instead of rebuilding a button. Extend with a spread, don't fork.
04Hover/focus needs a class hook. Inline styles can't express :hover. Add the matching uiClass.* (e.g. className={uiClass.btnPrimary}) so the rule in index.css applies. Form-control focus rings are global — no class needed.
05One dominant action per decision region. A page may contain several workflows, but locally competing primary actions are not allowed. Brand orange supports identity and indication; accessible action roles carry interactive text and fills.
06Status colour is functional, not brand. Use success/error/warning/info only for state, and apply it as an accent — dot, icon, left-border, or dark text — on a neutral surface. Avoid coloured fills; reserve the soft-tint tokens for rare emphasis.
07Stay on the scale and use its roles. A 4px base underpins spacing, but page, section, field, toolbar, and table-density roles determine which step is appropriate.
08Compact, readable type. Regular 400 + Semibold 600 carry product text; Medium 500 is reserved for controls. Normal UI copy is 15px, supporting and data text is 13–14px, and 11px labels remain short, high-contrast, and exceptional.
09Evidence before AI plumbing. Financial data, checks, source evidence, and filing state lead. Models, agents, tokens, and tool calls live in advanced or activity views.
10Choose the lightest structure. Use alignment, type, spacing, and dividers before adding a bordered group or card. Static surfaces stay flat; elevation is reserved for overlap.

How to apply

The mechanical pattern. This is what "use the design system" looks like in a React component.

A button

// import semantic tokens + shared component primitives
import { tokens } from "../lib/theme";
import { ui, uiClass } from "../lib/uiStyles";

// spread the variant; add the uiClass for the hover rule in index.css
<button style={ui.buttonPrimary} className={uiClass.btnPrimary}>
  Run extraction
</button>

// size modifier = spread after the variant
<button style={{ ...ui.buttonSecondary, ...ui.buttonSm }} className={uiClass.btnSecondary}>
  Cancel
</button>

A one-off element from semantic roles

// when no primitive fits, compose by purpose — never visual resemblance
<div style={{
  padding: tokens.space.inset,
  background: tokens.surface.default,
  border: `1px solid ${tokens.color.border.subtle}`,
  borderRadius: tokens.radius.panel,
  color: tokens.color.text.primary,
}}>…</div>
Do

Choose color.action.primary for a primary action · spread ui.card for a distinct object · add an interaction hook only when the object is navigable.

Don't

className="bg-orange-500" · style={{background:'#FD5108'}} · re-deriving a button's padding by hand.

Design principles

Six ideas behind every decision.

Evidence first

Financial data and its source are the primary visual objects. AI plumbing stays backstage.

Light & open

White surfaces, soft borders, generous spacing. Weight comes from space, not heavy fills.

Restrained type

Two weights — regular and semibold — carry the interface. Size, not many weights, creates hierarchy.

Orange with intent

Signature orange expresses identity and indication; accessible semantic roles carry actions and readable status text.

Quietly consistent

One compact type rhythm, one spacing system, restrained radii, and flat surfaces. Predictability is the feature.

Density follows the task

Setup flows breathe; review tables stay compact, aligned, and precise.

Functional colour — optional and exceptional

Routine status does not require colour. Prefer explicit text and neutral symbols. When functional colour materially improves urgency or differentiation, confine it to an alert icon, attention rule, chart mark, or rare highlighted row. Darker *Text roles carry any readable coloured label.
Soft tints still exist, but are reserved for rare emphasis (e.g. highlighting one reconciliation row) and are intentionally desaturated — close to neutral, not vivid. They are never the default status or alert surface.

Semantic token architecture

Values, meanings, and component decisions are separate layers. This keeps the product accessible and lets the visual identity evolve without search-and-replace styling.

1 · Global values

Raw ingredients such as orange500, grey700, spacing steps, radius values, and motion durations.

2 · Semantic roles

Purpose-led aliases such as color.text.secondary, color.action.primary, surface.canvas, and space.section.

3 · Component roles

Stable decisions such as button.primary.background.hover, table.header.surface, and dialog.scrim.

Selection rule: choose a token because its name matches the purpose, never because its current hex or pixel value happens to look right.

Elevation

Depth communicates overlap, not importance. Ordinary sections and cards are flat; sticky regions, menus, and dialogs use restrained shadows only when they sit above another surface.

Layouts & density

Width and density follow the job. Do not centre every workflow in the same card or spread every form across the viewport.

L1Form page. A constrained readable column for setup and account forms. Use ui.pageForm.
L2List page. A wide table or object list with sticky filters and stable columns. Use ui.pageWide.
L3Run report. Full-width outcome header, section tabs, then task-specific content. Readiness comes before telemetry.
L4Review workspace. Collapsible document navigation, financial grid, and source evidence panes. Preserve working context when panes change.
L5Grouping. Prefer section spacing, then a bordered group, then a static card. Use an interactive card only for a selectable or navigable object.
Responsive contract: wide screens show the complete review workspace; laptops may collapse side panes; narrow screens retain monitoring and simple actions rather than pretending the financial grid is comfortably editable.
Density contract: setup and explanation surfaces use comfortable spacing; operational lists use standard density; financial grids offer compact and standard modes. Density changes row height and padding, never typography contrast or target accessibility.
Viewport modes: mobile = monitor, approve, read evidence, and take simple actions · tablet = simplified grid with one contextual pane · laptop = primary review workspace · wide desktop = full grid plus navigation and evidence panes.

Motion

Motion explains state or spatial continuity. It is never decoration and never competes with financial review.

M1Use only pwc.motion.duration and pwc.motion.easing.
M2Prefer opacity and transform; avoid layout-shifting animation.
M3Do not animate financial values while they are being reviewed or edited.
M4Only interactive cards lift. Static groups and metric tiles remain still.
M5Every transition has an immediate equivalent under prefers-reduced-motion.

Buttons

One dominant action per decision region. Primary actions use an accessible dark-orange role; supporting actions remain neutral. Destructive actions stay quiet in ordinary contexts and become explicit at final confirmation.

Variants — ui.button*

Sizes — ui.buttonSm / buttonLg

// Four purposeful variants. Default target 44px; compact target 40px.
<button style={ui.buttonPrimary} className={uiClass.btnPrimary}>Run</button>
<button style={ui.buttonDanger}  className={uiClass.btnDanger}>Stop all</button>
Variant rule: Primary = commits the local workflow · Secondary = alternate or reversible action · Quiet = low-priority toolbar/navigation action · Destructive = names an irreversible consequence. Do not introduce a fifth visual variant to solve spacing or hierarchy.

Form controls

Readable labels, perceptible control boundaries, and a two-part focus treatment prioritize completion over decoration. Controls are 44px minimum; help and recovery copy is programmatically associated.

Remove the slash from the output name.
// Labels, help, and recovery are part of the component contract.
<label htmlFor="model" style={ui.fieldLabel}>Model</label>
<input id="model" style={ui.input} aria-describedby="model-help" />
Choice rule: Checkbox = include or select an option before submission. Switch = an immediate on/off setting whose effect happens when changed. Never use a switch merely because it looks more modern.

Status

Routine status uses a neutral symbol plus explicit text. Shape improves scanning, text carries the meaning, and no status hue competes with financial data or the primary action.

Status hierarchy: routine status stays monochrome · ordinary metadata remains plain text · actionable risk expands into an alert or attention row with recovery guidance · functional color is optional and confined to that exceptional treatment.

Cards

A card represents a distinct object. It is not the default wrapper for a section or metric. Static groups remain flat; only selectable or navigable cards receive a border/surface interaction treatment.

Grouping order: alignment and spacing → divider → inset surface → bordered object → interactive card. Stop at the lightest structure that makes the relationship clear. Never nest a card solely to create padding.

Tabs & navigation

The app shell uses horizontal destination links; run detail uses tabs for alternate views of one resource. The active destination is communicated by dark text plus an orange indicator, so brand color is not the only readable signal.

Run-detail sections

Semantics: destinations are links with stable URLs and aria-current. Tabs switch views of the same resource and use role="tablist" / role="tab" / role="tabpanel", roving tabindex, and arrow navigation. AI activity, models, token use, and traces belong under Technical details rather than competing with review outcomes.

Accessibility

The non-negotiable product floor. Each component documents how it meets these requirements; accessibility is not delegated to this section alone.

AVisible focus. Use a two-part indicator that remains perceptible on white, grey, tinted, and high-contrast surfaces. A soft halo may support focus but never carries it alone.
BARIA tab pattern. Navs and section switchers use role="tablist"/"tab"/"tabpanel" with aria-selected and arrow-key roving tabindex.
CColor is never the only signal. Status always includes explicit text and may add a neutral symbol or structural treatment. Color is optional, never required to understand the state.
DContrast is encoded in roles. Normal text meets 4.5:1; large text and essential control boundaries meet 3:1; disabled and decorative content are the only low-contrast exceptions.
EHit targets. Inputs and important actions are 44px minimum. Compact desktop controls may be 40px when separated; no interactive target falls below the WCAG 24px minimum.
FDecorative SVG is hidden. Brand/decorative marks carry aria-hidden="true"; meaningful icons get a label.
GErrors are associated. Inline help and validation messages are connected to their control with aria-describedby; the message explains recovery.
HDynamic work is announced. Upload, validation, save, filter-result, and run-progress changes use an appropriately polite live region.
IFocus survives transitions. Dialogs trap and return focus; sticky regions never obscure the focused control; dense grids provide a documented keyboard path.
JZoom and motion are tested. Core tasks remain usable at 200% zoom, and reduced motion reaches the same final state immediately.
KReflow and high contrast. Monitoring and simple actions work at 320 CSS px; forced-colors and high-contrast modes preserve boundaries, focus, and state meaning.
LDense grids have a keyboard model. Document navigation between cells, evidence, edit mode, validation, and frozen regions; do not rely on pointer hover.

Content & component states

Visual consistency includes language and feedback. Components are incomplete until their non-default states are designed.

C1Lead with the accounting task or consequence. Describe the discrepancy and the next action; keep internal check names and AI implementation detail secondary.
C2Use verbs for actions and nouns for destinations. Avoid friendly filler, generic AI-product language, and duplicate helper paragraphs.
C3State risk directly. Destructive and filing-risk actions name what happens, what remains, and how the user can recover.
C4Design the whole state set. Specify default, hover, focus, active, disabled, busy, success, error, and empty behaviour where relevant.
C5Reuse vocabulary. Repeated product terms live in the shared vocabulary helpers so labels do not drift across pages.
Required product patterns: page header, toolbar, sticky action bar, file drop zone, filter bar, empty state, loading/skeleton, toast, confirm dialog, disclosure, icon button, overflow actions, load-more, split pane, save state, attention queue, and evidence inspector.

Definition of a complete component

A visual example is not a component specification. Every production component documents the following before it is considered stable.

  1. Purpose and user outcome
  2. When to use and when not to use
  3. Anatomy and required parts
  4. Variants and density
  5. Default, hover, focus, active, selected, disabled, busy, success, error, and empty states
  6. Interaction and keyboard behaviour
  7. Responsive and overflow behaviour
  8. Accessibility requirements
  9. Content and vocabulary guidance
  10. Semantic and component token mapping
  11. Examples and anti-patterns
  12. Maturity: draft, beta, stable, or deprecated
Acceptance rule: a component is not stable until its non-default states, keyboard path, narrow-screen behaviour, and accessibility checks are demonstrated and tested.
