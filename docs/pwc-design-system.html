XBRL Agent Design System

The behavioural and visual specification for the web UI. Built on PwC's black-and-orange brand foundation — restrained typography, light surfaces, disciplined accent use. Production values live in web/src/lib/theme.ts, web/src/lib/uiStyles.ts, and web/src/index.css; parity tests keep specification and implementation aligned.

Overview

This document has two jobs: define the visual language (Foundations + Components), and tell AI developers exactly how to use it in this codebase. Because the app forbids CSS frameworks (gotcha #7), the components here are documented with the inline-style + token-object pattern the app actually uses, not CSS classes.

Governance: this document owns behavioural and visual rules; theme.ts, uiStyles.ts, and index.css own production implementation; tests own parity. A rule or value change updates all three layers in one change set. The CSS variables here exist only to render this reference.

Rules for AI developers

The load-bearing conventions. Adhering to the design system means following these — not just picking nice colors.

01Inline values; narrow state hooks. Layout, colour, type, spacing, and geometry use inline style={{…}} objects. Do not use Tailwind or a class-based utility system. A shared uiClass.* hook is allowed only for hover, focus, animation, editor styling, or another state inline styles cannot express.
02Tokens, never literals. Import { pwc } from lib/theme and reference pwc.orange500, pwc.grey200, etc. Never hard-code a hex, px radius, or font stack.
03Reuse the primitives. Spread ui.buttonPrimary, ui.card, ui.input… from lib/uiStyles instead of rebuilding a button. Extend with a spread, don't fork.
04Hover/focus needs a class hook. Inline styles can't express :hover. Add the matching uiClass.* (e.g. className={uiClass.btnPrimary}) so the rule in index.css applies. Form-control focus rings are global — no class needed.
05One primary action per view. Orange marks the single most important action or status. Everything else is neutral. Orange is never decoration.
06Status colour is functional, not brand. Use success/error/warning/info only for state, and apply it as an accent — dot, icon, left-border, or dark text — on a neutral surface. Avoid coloured fills; reserve the soft-tint tokens for rare emphasis.
07Stay on the scale. Spacing from pwc.space (4px base), radius from pwc.radius, shadow from pwc.shadow. No arbitrary values.
08Two text weights. Regular 400 + Semibold 600 carry all text (Medium 500 only on controls). No Light 300 — it reads as a different typeface — and never 700+. Hierarchy comes from size, not a wide weight range.
09Evidence before AI plumbing. Financial data, checks, source evidence, and filing state lead. Models, agents, tokens, and tool calls live in advanced or activity views.
10Choose the lightest structure. Use alignment, type, spacing, and dividers before adding a bordered group or card. Elevation signals interactivity, not decoration.

How to apply

The mechanical pattern. This is what "use the design system" looks like in a React component.

A button

// import the token object + the shared primitives
import { pwc } from "../lib/theme";
import { ui, uiClass } from "../lib/uiStyles";

// spread the variant; add the uiClass for the hover rule in index.css
<button style={ui.buttonPrimary} className={uiClass.btnPrimary}>
  Run extraction
</button>

// size modifier = spread after the variant
<button style={{ ...ui.buttonSecondary, ...ui.buttonSm }} className={uiClass.btnSecondary}>
  Cancel
</button>

A one-off element from raw tokens

// when no primitive fits, build inline from pwc.* — never literals
<div style={{
  padding: pwc.space.lg,
  background: pwc.white,
  border: `1px solid ${pwc.grey200}`,
  borderRadius: pwc.radius.lg,
  boxShadow: pwc.shadow.card,
  color: pwc.grey900,
}}>…</div>
Do

background: pwc.orange500 · spread ui.card · add uiClass.card for the hover lift.

Don't

className="bg-orange-500" · style={{background:'#FD5108'}} · re-deriving a button's padding by hand.

Design principles

Four ideas behind every decision.

Evidence first

Financial data and its source are the primary visual objects. AI plumbing stays backstage.

Light & open

White surfaces, soft borders, generous spacing. Weight comes from space, not heavy fills.

Restrained type

Two weights — regular and semibold — carry the interface. Size, not many weights, creates hierarchy.

Orange with intent

The signature orange marks the single most important action or status in a view — never decoration.

Quietly consistent

One radius scale, one spacing rhythm, one shadow family. Predictability is the feature.

Density follows the task

Setup flows breathe; review tables stay compact, aligned, and precise.

Color

Orange leads as the single signature accent. White carries backgrounds; greys handle structure. The status hues are a muted, warm-neutral family that sits quietly beside the orange — applied as dots, icons, left-borders and dark text, not as coloured fills.

Status — applied as accents, not fills

Status colour is an accent, not a surface. Carry it with a coloured dot, icon, left-border, or dark text on a neutral surface — never a coloured background. This keeps the UI calm and lets the orange stay the loudest colour in any view. Bright base hues drive the small accents — dots, icons, left-borders. The darker *Text tokens are for the rare bit of coloured status text that needs contrast (card deltas, inline messages, do/don't headers) — successText #157A53 · errorText #C0303A · warningText #8A6111 · infoText #2C6299. Badge labels stay neutral.
Soft tints still exist, but are reserved for rare emphasis (e.g. highlighting one reconciliation row) and are intentionally desaturated — close to neutral, not vivid. They are no longer the default surface for badges or alerts.

Typography

A single Helvetica Neue family (pwc.fontHeading = pwc.fontBody) and a deliberately short scale — five sizes, two text weights. Size does the work: Regular 400 for body and data, Semibold 600 for titles, headings, emphasis and labels. There is no light weight — a wide weight jump reads as a different typeface.

Text weights: Regular 400 (body, data) + Semibold 600 (titles, headings, emphasis, labels). Medium 500 appears only on interactive controls (buttons, tabs, nav); Light 300 is not used (it read as a different typeface) and never 700+. UI primitives keep fixed control sizes (button/input 15, field label 14, badge 12) — component constants, not extra scale steps.

Layouts & density

Width and density follow the job. Do not centre every workflow in the same card or spread every form across the viewport.

L1Form page. A constrained readable column for setup and account forms. Use ui.pageForm.
L2List page. A wide table or object list with sticky filters and stable columns. Use ui.pageWide.
L3Run report. Full-width outcome header, section tabs, then task-specific content. Readiness comes before telemetry.
L4Review workspace. Collapsible document navigation, financial grid, and source evidence panes. Preserve working context when panes change.
L5Grouping. Prefer section spacing, then a bordered group, then a static card. Use an interactive card only for a selectable or navigable object.
Responsive contract: wide screens show the complete review workspace; laptops may collapse side panes; narrow screens retain monitoring and simple actions rather than pretending the financial grid is comfortably editable.

Motion

Motion explains state or spatial continuity. It is never decoration and never competes with financial review.

M1Use only pwc.motion.duration and pwc.motion.easing.
M2Prefer opacity and transform; avoid layout-shifting animation.
M3Do not animate financial values while they are being reviewed or edited.
M4Only interactive cards lift. Static groups and metric tiles remain still.
M5Every transition has an immediate equivalent under prefers-reduced-motion.

Buttons

One primary action per view, carried by orange. Everything else recedes to neutral. Destructive actions stay quiet (outline) until hovered.

Variants — ui.button*

Sizes — ui.buttonSm / buttonLg

<button style={ui.buttonDanger}  className={uiClass.btnDanger}>Stop all</button>
Pairing rule: every button needs both style={ui.buttonX} (geometry + color) and className={uiClass.btnX} (the hover/disabled rule in index.css). Without the class, the button has no hover state.

Badges & status

Use the quietest treatment that communicates the state. Ordinary metadata stays plain; routine status uses a dot and text; outline badges are reserved for compact, high-value state identification; actionable risk uses an alert.

Do not badge ordinary metadata. Filing standard, filing level, denomination, dates and model names normally render as aligned text. A badge earns its border only when the state must be identified at a glance.

Tabs & navigation

The app shell is a horizontal top-nav (not a sidebar — that's just this doc's chrome). Underline tabs, orange for the active item. Run-detail uses the same pattern for its section tabs.

Top nav — TopNav.tsx

Run-detail sections

Semantics: top-level destinations are navigation and must expose stable URLs. Run-detail section switchers use role="tablist" / role="tab" with aria-selected, roving tabindex, and arrow navigation. Do not claim keyboard behaviour until it is implemented and tested.

Accessibility

The conventions the app already implements — keep them when adding UI.

AVisible focus. Keyboard focus shows a 2px orange outline on buttons (:focus-visible in index.css) and a 3px orange-50 ring on form controls. Never remove these.
BARIA tab pattern. Navs and section switchers use role="tablist"/"tab"/"tabpanel" with aria-selected and arrow-key roving tabindex.
CColor is never the only signal. Status badges carry a text label (and a dot), not just a hue. Don't encode meaning in color alone.
DStatus text contrast. Use the darker *Text tokens (successText/errorText/warningText/infoText) for status labels so they stay AA-legible on neutral surfaces; keep alert body copy at grey800.
EHit targets. Inputs/selects are 44px min-height, buttons 40px. Labels are real <label> elements tied to their control.
FDecorative SVG is hidden. Brand/decorative marks carry aria-hidden="true"; meaningful icons get a label.
GErrors are associated. Inline help and validation messages are connected to their control with aria-describedby; the message explains recovery.
HDynamic work is announced. Upload, validation, save, filter-result, and run-progress changes use an appropriately polite live region.
IFocus survives transitions. Dialogs trap and return focus; sticky regions never obscure the focused control; dense grids provide a documented keyboard path.
JZoom and motion are tested. Core tasks remain usable at 200% zoom, and reduced motion reaches the same final state immediately.

Content & component states

Visual consistency includes language and feedback. Components are incomplete until their non-default states are designed.

C1Lead with the accounting task or consequence. Describe the discrepancy and the next action; keep internal check names and AI implementation detail secondary.
C2Use verbs for actions and nouns for destinations. Avoid friendly filler, generic AI-product language, and duplicate helper paragraphs.
C3State risk directly. Destructive and filing-risk actions name what happens, what remains, and how the user can recover.
C4Design the whole state set. Specify default, hover, focus, active, disabled, busy, success, error, and empty behaviour where relevant.
C5Reuse vocabulary. Repeated product terms live in the shared vocabulary helpers so labels do not drift across pages.
Required product patterns: page header, toolbar, sticky action bar, file drop zone, filter bar, empty state, loading/skeleton, toast, confirm dialog, disclosure, icon button, overflow actions, load-more, split pane, save state, attention queue, and evidence inspector.
